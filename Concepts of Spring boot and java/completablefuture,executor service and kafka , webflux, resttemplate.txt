====================================================
COMPLETABLEFUTURE - REAL-WORLD SCENARIOS
====================================================

WHY IT'S USED (ONE LINE)
----------------------------------------------------
   Whenever an application needs to run multiple 
   slow, I/O-heavy operations (API calls, DB 
   queries, file/network operations) WITHOUT 
   blocking threads while waiting.


BASIC / COMMON SCENARIOS
----------------------------------------------------

1. Microservices Aggregator (BFF Pattern)
   A single API gateway or backend-for-frontend 
   service needs to call 3-4 downstream 
   microservices (inventory, pricing, user 
   profile, reviews) to build one combined 
   response. Calling them one after another wastes 
   time; running them in parallel and combining 
   results cuts total response time drastically.

2. Fraud/Risk/Compliance Checks (FinTech/Banking)
   Before approving a transaction, the system 
   must run fraud detection, credit limit check, 
   and AML (anti-money-laundering) check -- all 
   independent, all calling external services. All 
   three must complete before a decision is made.

3. Shipment/Order Tracking Aggregation (Logistics)
   A tracking dashboard pulls live status from 
   multiple courier partner APIs (FedEx, DHL, 
   local carriers) at the same time and merges 
   them into one unified view for the customer.

4. Sending Notifications After a Core Action
   After a payment or order is completed, the 
   system needs to send an email, SMS, and push 
   notification. The customer shouldn't wait for 
   all three to finish -- the main response 
   returns immediately while notifications happen 
   in the background.

5. Report Generation / Dashboard Loading
   A dashboard loads sales data, user stats, and 
   analytics graphs from different data sources. 
   Each panel loads independently and 
   asynchronously instead of the whole page 
   waiting on the slowest query.

6. Calling an External Payment Gateway With a 
   Timeout
   When charging a card through a third-party 
   payment gateway that occasionally hangs, the 
   system needs to fail fast after a few seconds 
   and fall back to a retry or error message, 
   instead of blocking indefinitely.


COMPLEX / ADVANCED SCENARIOS
----------------------------------------------------

7. Multi-Level Dependent Workflow (Chained Calls)
   Scenario: An onboarding flow where you first 
   fetch a customer's KYC (identity verification) 
   status, THEN based on that result fetch their 
   credit bureau score, THEN based on that decide 
   the loan eligibility, THEN generate a loan 
   offer document. Each step depends on the 
   previous one's result, but none of them should 
   block a thread while waiting -- they get 
   chained so each step kicks off automatically 
   once the prior one completes.

8. Partial Failure Handling in a Fan-Out Call
   Scenario: A travel booking system calls 5 
   different airline/hotel partner APIs 
   simultaneously to fetch prices. Some partners 
   might time out or fail. The system must still 
   return whatever results succeeded within a 
   time budget (e.g. 2 seconds), gracefully 
   ignoring the failed/slow ones rather than 
   letting one bad partner block the entire 
   search results page.

9. Combining Two Independent Async Results Into 
   One Business Decision
   Scenario: A loan approval engine needs BOTH the 
   applicant's income verification (from a payroll 
   API) AND their existing debt information (from 
   a credit bureau API) before it can calculate a 
   debt-to-income ratio. Both calls run in 
   parallel, and only once BOTH finish does the 
   system combine them into a single approve/
   reject decision.

10. Retry With Fallback Chains (Resilience 
    Pattern)
    Scenario: A logistics system tries to fetch 
    real-time delivery ETA from a primary GPS 
    tracking provider. If that call fails or times 
    out, it automatically falls back to a 
    secondary, less accurate provider, and if that 
    also fails, falls back to a cached/estimated 
    ETA -- all without blocking the customer-facing 
    API response.

11. Batch Processing With Controlled Concurrency
    Scenario: A nightly batch job needs to process 
    10,000 customer records (e.g. sending 
    personalized statements or recalculating 
    interest for each account). Instead of 
    processing one at a time (too slow) or all 
    10,000 at once (overwhelms the system/DB), 
    tasks are submitted in controlled batches 
    using a custom thread pool, and the job 
    tracks overall completion across the entire 
    batch.

12. Circuit-Breaker-Style Orchestration in a 
    Payment Pipeline
    Scenario: A payment processing pipeline calls: 
    inventory reservation -> payment gateway 
    charge -> shipping label creation -> 
    notification, where some steps are independent 
    (can run in parallel) and some are dependent 
    (must happen in sequence). If the payment step 
    fails, the system needs to automatically 
    trigger a compensating action (release the 
    inventory reservation) -- coordinating a mix 
    of parallel and sequential async steps with 
    failure recovery baked in.

13. Real-Time Data Enrichment Before a UI Response
    Scenario: A customer support agent's screen 
    needs to load a customer's profile enriched 
    with data from 6 different systems (CRM, 
    billing, past tickets, loyalty points, 
    account status, recent transactions) the 
    moment they open a ticket. All 6 lookups run 
    concurrently, and the page renders 
    progressively as each piece of data arrives, 
    rather than waiting for the slowest system.


REMEMBER THIS LINE
----------------------------------------------------
   Simple scenarios = run independent slow calls 
   in PARALLEL instead of one-by-one.
   Complex scenarios = combine PARALLEL work, 
   SEQUENTIAL dependencies, PARTIAL failures, 
   TIMEOUTS, and FALLBACKS all in one coordinated 
   async workflow -- without ever blocking a 
   thread while waiting.

====================================================
EXECUTORSERVICE - REAL-WORLD SCENARIOS
====================================================

WHY IT'S USED (ONE LINE)
----------------------------------------------------
   Whenever you need a managed pool of threads to 
   run tasks concurrently, without manually 
   creating/destroying Thread objects yourself.


BASIC / COMMON SCENARIOS
----------------------------------------------------

1. Background File Processing
   An upload service receives a large CSV/Excel 
   file and needs to parse and insert thousands of 
   rows into the DB. Instead of blocking the 
   upload request, the parsing job is submitted to 
   a thread pool and processed in the background.

2. Sending Bulk Emails/SMS
   A marketing service needs to send promotional 
   emails to 50,000 users. Each email send is 
   submitted as a task to a fixed thread pool so 
   multiple emails go out concurrently instead of 
   one at a time.

3. Scheduled/Periodic Jobs
   A logistics system needs to check for delayed 
   shipments every 5 minutes and flag them. 
   ScheduledExecutorService runs this recurring 
   job in the background without a dedicated 
   always-running thread.

4. Parallel Batch DB Updates
   A banking system needs to recalculate interest 
   for 10,000 accounts overnight. The workload is 
   split into chunks and each chunk is submitted 
   to the executor's thread pool to run in 
   parallel.

5. Web Server Request Handling (Under the Hood)
   Traditional servlet containers (like Tomcat) 
   use a thread pool (backed by ExecutorService 
   concepts) to handle incoming HTTP requests -- 
   each request gets a worker thread from the pool.


COMPLEX / ADVANCED SCENARIOS
----------------------------------------------------

6. Controlled Concurrency to Avoid Overloading 
   Downstream Systems
   Scenario: A logistics system needs to call a 
   slow external customs-clearance API for 5,000 
   shipments. Instead of firing all 5,000 requests 
   at once (which would crash the partner's 
   system), a fixed thread pool of, say, 20 threads 
   throttles how many calls happen simultaneously, 
   protecting both systems.

7. Graceful Shutdown During Deployment
   Scenario: A payment service is mid-way through 
   processing pending transactions when a new 
   deployment needs to roll out. The executor must 
   stop accepting new tasks (shutdown()), but 
   finish in-flight transactions before the JVM 
   terminates, to avoid partially processed 
   payments.

8. Isolating Thread Pools per Downstream Dependency 
   (Bulkhead Pattern)
   Scenario: A single microservice calls three 
   downstream systems: a fast internal cache, a 
   medium-speed internal DB, and a slow third-party 
   credit bureau API. Using ONE shared thread pool 
   for all three means a slow credit bureau call 
   can exhaust all threads and starve the fast 
   internal calls too. Separate, isolated thread 
   pools per dependency prevent one slow system 
   from taking down the whole service.

====================================================
KAFKA - REAL-WORLD SCENARIOS
====================================================

WHY IT'S USED (ONE LINE)
----------------------------------------------------
   Whenever systems need to communicate 
   asynchronously and reliably at scale, without 
   being tightly coupled or waiting on each other 
   directly.


BASIC / COMMON SCENARIOS
----------------------------------------------------

1. Order Processing Pipeline (E-commerce/Logistics)
   When a customer places an order, an "OrderPlaced" 
   event is published to Kafka. Separate services 
   (inventory, billing, shipping, notifications) 
   each consume this event independently and do 
   their part, without the order service needing 
   to know or wait for them.

2. Real-Time Fraud Detection (FinTech)
   Every transaction event is streamed into Kafka. 
   A fraud detection service consumes this stream 
   in real time, scoring each transaction and 
   flagging suspicious ones instantly, without 
   slowing down the main payment flow.

3. Log/Event Aggregation
   Multiple microservices publish their logs or 
   audit events to a Kafka topic, and a central 
   monitoring/analytics service consumes them to 
   build dashboards and alerts.

4. Decoupling Microservices
   Instead of Service A directly calling Service B 
   (tight coupling, B must be up), Service A 
   publishes an event to Kafka and Service B 
   consumes it whenever it's ready -- both can be 
   deployed, scaled, or restarted independently.

5. Notification Systems
   A "PaymentCompleted" event triggers separate 
   Kafka consumers for email, SMS, and push 
   notification services, each processing the 
   event on their own schedule.


COMPLEX / ADVANCED SCENARIOS
----------------------------------------------------

6. Event Sourcing / Audit Trail for Regulatory 
   Compliance (Banking)
   Scenario: A bank needs a complete, replayable 
   history of every account state change for 
   regulatory audits. Instead of just storing 
   current state in a DB, every change is published 
   as an immutable event to Kafka. The current 
   account balance can be rebuilt at any time by 
   replaying all events from the beginning -- 
   useful for audits, debugging, and disaster 
   recovery.

7. Exactly-Once Processing for Financial 
   Transactions
   Scenario: A payment must be processed EXACTLY 
   once -- not zero times (money lost) and not 
   twice (double charge). Using Kafka's idempotent 
   producers and transactional consumers, the 
   system guarantees a transaction event is 
   neither lost nor duplicated, even if a service 
   crashes and restarts mid-processing.

8. Multi-Region Data Replication with Ordering 
   Guarantees
   Scenario: A logistics company operates across 
   multiple countries, each with its own regional 
   database, but needs a unified, ordered view of 
   all shipment status changes globally for a 
   central dashboard. Kafka's partitioning (keyed 
   by shipment ID) ensures all events for the same 
   shipment stay in order, even when produced from 
   different regions.

9. Handling Consumer Lag During Traffic Spikes
   Scenario: During a flash sale, order events 
   spike 50x normal volume. Consumers can't keep 
   up, causing lag. The system needs partition 
   scaling, consumer group rebalancing, and 
   backpressure handling so the system degrades 
   gracefully (delayed but eventually consistent 
   processing) instead of crashing or dropping 
   events.

10. Saga Pattern for Distributed Transactions
    Scenario: Placing an order involves multiple 
    services: reserve inventory -> charge payment 
    -> create shipment. There's no single database 
    transaction across all three. Kafka events 
    coordinate this as a "saga" -- if the payment 
    step fails, a compensating "ReleaseInventory" 
    event is published to undo the earlier step, 
    keeping the whole system eventually consistent 
    without a distributed lock.

====================================================
SPRING WEBFLUX - REAL-WORLD SCENARIOS
====================================================

WHY IT'S USED (ONE LINE)
----------------------------------------------------
   Whenever a system needs to handle a very high 
   number of concurrent requests efficiently using 
   a small number of threads, especially when 
   those requests involve waiting on slow I/O.


BASIC / COMMON SCENARIOS
----------------------------------------------------

1. High-Traffic Public APIs
   A flight/hotel search API that receives 
   thousands of concurrent requests per second, 
   each waiting on multiple downstream partner API 
   calls, uses WebFlux to avoid tying up one 
   thread per request while waiting.

2. Streaming Live Data to Clients
   A stock ticker or live shipment tracking 
   dashboard streams continuous updates to the 
   browser using Server-Sent Events (SSE) or 
   WebSockets, backed by WebFlux's reactive 
   streams.

3. Aggregating Multiple Backend Calls Non-Blockingly
   A product page needs to call inventory, 
   pricing, and reviews services and combine them 
   -- WebFlux allows this entire chain to be 
   non-blocking end-to-end, unlike blocking REST 
   controllers.

4. Chat Applications / Real-Time Notifications
   A messaging app needs to push real-time 
   notifications to thousands of connected users 
   simultaneously, which WebFlux handles well 
   using a small, fixed thread pool (event-loop 
   model).


COMPLEX / ADVANCED SCENARIOS
----------------------------------------------------

5. High-Concurrency Payment Gateway Under Load
   Scenario: A payment gateway must handle 
   100,000+ concurrent requests during a sale 
   event, where each request calls 3-4 downstream 
   services (fraud check, bank API, ledger 
   update). Using a traditional thread-per-request 
   model would need tens of thousands of threads 
   (expensive). WebFlux handles this with a small 
   number of event-loop threads, since threads are 
   never blocked waiting on I/O -- they're freed up 
   the moment a call is made and reused for other 
   requests while waiting for the response.

6. Backpressure Handling in IoT/Sensor Data 
   Ingestion (Logistics)
   Scenario: Thousands of GPS trackers on delivery 
   trucks send location updates every few seconds. 
   If the backend can't process updates fast 
   enough, it needs to slow down or buffer 
   producers gracefully rather than crash from 
   being overwhelmed. WebFlux's reactive streams 
   have built-in backpressure -- the consumer 
   signals how much data it can handle, and the 
   producer respects that pace.

7. Composing Multiple Reactive Data Sources with 
   Fallbacks
   Scenario: A dashboard needs data from a 
   reactive database (R2DBC), a reactive cache 
   (Redis reactive), and a downstream reactive 
   REST API -- combined into one non-blocking 
   response stream. If the cache is empty, it 
   falls back to the DB reactively; if the REST 
   call times out, it falls back to a cached 
   value -- all composed as a single reactive 
   pipeline without ever blocking a thread.

8. Migrating a Legacy Blocking System Incrementally
   Scenario: A large FinTech platform wants to 
   migrate from blocking Spring MVC + RestTemplate 
   to WebFlux, but can't rewrite everything at 
   once. Certain blocking legacy calls (like JDBC) 
   need to be wrapped carefully (using 
   Schedulers.boundedElastic()) so they don't 
   block the small number of WebFlux event-loop 
   threads, while still allowing gradual, 
   service-by-service migration.

====================================================
RESTTEMPLATE - REAL-WORLD SCENARIOS
====================================================

WHY IT'S USED (ONE LINE)
----------------------------------------------------
   For making synchronous, blocking HTTP calls to 
   other services -- simple, well-understood, and 
   still widely used in traditional (non-reactive) 
   Spring Boot microservices.

NOTE: RestTemplate is now in maintenance mode; 
Spring recommends WebClient for new projects. 
Still very common in existing/legacy codebases 
and interviews.


BASIC / COMMON SCENARIOS
----------------------------------------------------

1. Calling Another Internal Microservice
   An Order Service needs to call the Inventory 
   Service synchronously to check stock 
   availability before confirming an order.

2. Integrating with Third-Party APIs
   A logistics system calls a courier partner's 
   REST API to fetch shipping rates or create a 
   shipment label.

3. Simple CRUD Client Calls
   A backend admin tool calls a user-management 
   microservice's REST endpoints to fetch, create, 
   or update user records.

4. Legacy Monolith-to-Microservice Communication
   An older monolithic banking application calls 
   newer microservices (e.g. a new KYC 
   verification service) via simple synchronous 
   REST calls during a gradual system migration.


COMPLEX / ADVANCED SCENARIOS
----------------------------------------------------

5. Combining RestTemplate with Resilience Patterns
   Scenario: A payment service calls a third-party 
   gateway using RestTemplate, but that gateway is 
   occasionally slow or down. The system wraps the 
   call with a circuit breaker (Resilience4j), 
   retry logic, and a fallback response -- so a 
   single failing dependency doesn't cascade and 
   take down the whole payment flow.

6. Thread Pool Exhaustion Under High Load
   Scenario: A high-traffic service uses 
   RestTemplate (blocking) to call 3 downstream 
   services per request. Under heavy load, all 
   available servlet threads get tied up waiting 
   on slow downstream responses, and the whole 
   service becomes unresponsive -- a classic 
   argument for eventually migrating such 
   high-concurrency paths to WebFlux/WebClient, 
   but a scenario interviewers love to ask about to 
   test understanding of blocking I/O limitations.

7. Chained Synchronous Calls Across Multiple 
   Services (Distributed Latency Stacking)
   Scenario: An order confirmation flow calls 
   Service A, which internally calls Service B 
   using RestTemplate, which internally calls 
   Service C -- each call blocking and waiting. 
   Latencies stack up (200ms + 200ms + 200ms = 
   600ms), and a slow deep dependency silently 
   degrades the entire chain's response time -- 
   this scenario is often used to explain why 
   async/reactive alternatives or careful timeout 
   configuration matters in deep microservice call 
   chains.

====================================================
RESTCLIENT - REAL-WORLD SCENARIOS
====================================================

WHAT IT IS (ONE LINE)
----------------------------------------------------
   RestClient is Spring's NEWER synchronous HTTP 
   client (introduced in Spring Framework 6.1 / 
   Boot 3.2). Same blocking behavior as 
   RestTemplate, but with a modern, fluent API 
   similar to WebClient -- WITHOUT going reactive.


WHY IT EXISTS
----------------------------------------------------
   - RestTemplate: old, synchronous, clunky API, 
     now in maintenance mode.
   - WebClient: modern, but reactive/non-blocking 
     -- overkill if you don't need reactive 
     programming and don't want Project Reactor 
     complexity.
   - RestClient: fills the gap -- synchronous 
     (easy to reason about) + fluent/modern API 
     (easy to write). Spring's recommended choice 
     for NEW non-reactive Spring MVC projects.


BASIC / COMMON SCENARIOS
----------------------------------------------------

1. Calling Internal Microservices in a Traditional 
   Spring MVC App
   A standard (non-WebFlux) Spring Boot 
   microservice needs to call another internal 
   service synchronously -- e.g. an Order Service 
   calling the Inventory Service to check stock -- 
   using a clean, modern fluent API instead of 
   RestTemplate's older verbose syntax.

2. Replacing RestTemplate in New Projects
   A team starting a brand-new Spring Boot 3.2+ 
   project that doesn't need reactive/WebFlux 
   picks RestClient by default, since Spring 
   explicitly recommends it over RestTemplate for 
   new synchronous applications.

3. Calling Third-Party REST APIs
   A logistics service calls a courier partner's 
   REST API (get shipping rate, create shipment 
   label) using RestClient's fluent builder syntax 
   -- easier to read and maintain than the older 
   RestTemplate calls.

4. Declarative HTTP Clients via @HttpExchange
   Instead of manually building requests, a team 
   defines a Java interface with @HttpExchange 
   annotations describing the API contract, and 
   Spring auto-generates the RestClient-backed 
   implementation -- common in projects wanting 
   Feign-client-like simplicity without adding 
   Feign as a dependency.

5. Migrating Legacy RestTemplate Code Gradually
   A team maintaining an older Spring Boot app 
   with lots of RestTemplate calls migrates 
   service-by-service to RestClient (since Spring 
   provides a near drop-in migration path), 
   without needing a full rewrite to WebClient/
   reactive.


COMPLEX / ADVANCED SCENARIOS
----------------------------------------------------

6. Migration Bridge in a Mixed Reactive/Non-Reactive 
   System (FinTech)
   Scenario: A banking platform has some services 
   fully migrated to WebFlux/WebClient, but 
   several core services still run classic Spring 
   MVC on servlet threads (e.g. legacy JDBC-based 
   ledger service). Rather than force WebClient's 
   reactive API into a blocking codebase (which 
   often causes confusing .block() calls and 
   subtle bugs), the team uses RestClient for 
   those services -- keeping the synchronous 
   mental model intact while still getting a 
   modern, maintainable API, as part of a phased 
   migration strategy.

7. Choosing RestClient vs WebClient Based on Load 
   Characteristics
   Scenario: An internal admin/reporting service 
   handles low, predictable traffic (a handful of 
   internal users), so the overhead of adopting 
   WebFlux + reactive programming isn't justified. 
   The team deliberately chooses RestClient over 
   WebClient here -- reserving WebClient only for 
   the customer-facing, high-concurrency 
   microservices -- as a case study in matching 
   the right HTTP client to actual traffic 
   patterns instead of blindly using "the newest" 
   tool everywhere.

8. Declarative HTTP Interfaces Shared Across Teams 
   (API Contract Standardization)
   Scenario: A large organization with dozens of 
   microservices wants every team calling a shared 
   internal "Customer Service" API to use the 
   exact same request/response contract, timeout 
   settings, and error handling. A shared 
   @HttpExchange interface backed by RestClient is 
   published as a common library, so every 
   consuming team gets a consistent, type-safe 
   client instead of each team hand-rolling their 
   own RestTemplate/WebClient calls with 
   inconsistent configurations.

9. Testing and Prototyping Before Committing to 
   Reactive Architecture
   Scenario: A team is unsure whether a new 
   service will eventually need high concurrency/
   reactive support. They start with RestClient 
   (synchronous, simple, blocking) to prototype 
   quickly and validate business logic, with the 
   option to swap to WebClient later if load 
   testing reveals the need for non-blocking I/O 
   -- since RestClient and WebClient share design 
   philosophy, the migration path later is 
   smoother than jumping from RestTemplate.


QUICK COMPARISON TABLE
----------------------------------------------------

  CLIENT          BLOCKING?    API STYLE       BEST FOR
  --------------------------------------------------------
  RestTemplate     Yes          Old/verbose     Legacy apps only
  RestClient       Yes          Fluent/modern   New sync Spring MVC apps
  WebClient        No (reactive) Fluent/reactive High-concurrency, WebFlux apps


REMEMBER THIS LINE
----------------------------------------------------
   RestTemplate = old and blocking.
   WebClient    = modern and non-blocking 
                  (reactive).
   RestClient   = modern API, but STILL blocking 
                  -- the "best of both worlds" 
                  choice when you want clean code 
                  without going fully reactive.

Simple idea
Eventual consistency = "it will be correct... just not immediately."
Imagine you update your Instagram bio. You see the new bio instantly (your phone). But your friend, on the other side of the world, might still see the old bio for a few seconds — because the update hasn't reached the server near her yet.
After a few seconds, her app also shows the new bio.
So:

❌ Not instant everywhere
✅ Eventually, everyone sees the same correct data

That's it. That's eventual consistency.
Compare with the opposite (Strong Consistency)
Bank ATM withdrawal: the moment you withdraw ₹5000, your balance must update immediately everywhere — no delay allowed. That's strong consistency — no room for "eventually."
Where you'll see this in your Spring Boot job
Example: E-commerce order

You click "Place Order" ✅ (Order saved in Order DB — instant)
Message sent to Kafka: "Hey, an order happened"
Inventory Service reads that message a few milliseconds later and reduces stock
Notification Service reads it a bit later and sends you an SMS

So right after placing the order:

Order status = CONFIRMED ✅ (immediate)
Stock count = still old value for a moment ⏳ (updates soon after)
SMS = not sent yet ⏳ (comes soon after)

Everything becomes correct eventually — just not all at the exact same millisecond.
One-line answer for interview

"Eventual consistency means different parts of the system don't update at the exact same moment, but if you wait a little, they all become correct and matching."

Short answer: you can't have true "same instant" consistency in a distributed/async system — that's the whole tradeoff. But you can force stronger consistency when you really need it, using different techniques. Here's how:
1. Switch that part to Synchronous call (not events)
Instead of Kafka (async), call the other service directly and wait for the response.
java@Transactional
public void placeOrder(Order order) {
    orderRepository.save(order);
    
    // Synchronous REST call - waits for inventory to actually update
    ResponseEntity<String> response = restTemplate.postForEntity(
        "http://inventory-service/reduce-stock", order, String.class);
    
    if (!response.getStatusCode().is2xxSuccessful()) {
        throw new RuntimeException("Stock update failed, rolling back order");
    }
}
Now both Order and Inventory update before the user gets a response — same moment, consistent. But you lose availability/speed if inventory service is slow/down.
2. Distributed Transaction (2-Phase Commit / XA)
Forces all databases to commit together or none at all. Rare in modern microservices (slow, tightly coupled) but exists via JtaTransactionManager in Spring.

3. Strong consistency within ONE database
If Order and Inventory are in the same DB, just use a normal @Transactional — this is instant, not eventual.
java@Transactional
public void placeOrder(Order order) {
    orderRepository.save(order);
    inventoryRepository.reduceStock(order.getProductId(), order.getQty());
    // Both succeed or both fail — same DB, same transaction, same moment
}
The real answer
WantUseFast, scalable, ok with slight delayEventual consistency (Kafka/events) — most microservices use thisMust be correct at the exact same instantSynchronous call / same DB transaction — use only for critical operations like payments
Rule of thumb (used in real companies)

Money/Payment → strong consistency (synchronous, same transaction)
Stock count, notifications, analytics, recommendations → eventual consistency (Kafka, async) — a few seconds delay is acceptable








How to Scale a REST API (Short)
1. Horizontal Scaling (Load Balancing)
Run multiple instances of your Spring Boot app behind a load balancer (Nginx/AWS ELB) — distributes traffic across servers.
2. Caching
Avoid hitting DB every time.
java@Cacheable(value = "products", key = "#id")
public Product getProduct(Long id) {
    return productRepository.findById(id).orElseThrow();
}
Use Redis/Memcached for frequently read, rarely changed data.
3. Database Scaling

Read Replicas → route reads to replica, writes to primary
Sharding → split data across multiple DBs by key (userId, region)
Connection pooling → HikariCP (default in Spring Boot)

4. Async / Non-blocking processing
Use Spring WebFlux (reactive) instead of blocking Spring MVC for high-concurrency APIs.
javapublic Mono<Product> getProduct(Long id) {
    return productRepository.findById(id);
}
5. Message Queues for heavy tasks
Offload slow work (emails, reports, notifications) to Kafka/RabbitMQ instead of doing it inline in the API call.
6. Rate Limiting / Throttling
Prevent abuse and protect backend — use API Gateway (Spring Cloud Gateway) or Bucket4j.
7. Pagination
Never return huge lists in one response.
javaPage<Product> products = productRepository.findAll(PageRequest.of(page, size));
8. CDN for static content
Offload images/JS/CSS from your API servers.
9. Stateless APIs
No session state on server (use JWT) → any instance can handle any request → easy horizontal scaling.
10. Microservices + API Gateway

Why Use Kafka in Spring Boot
Kafka = a distributed messaging/event-streaming platform used for async communication between microservices — instead of services calling each other directly (tightly coupled), they publish/consume events (loosely coupled).



KAFKA - ADVANTAGES IN SPRING BOOT

1. Decoupling
   Services don't call each other directly — no cascading failures

2. Async Processing
   Order API responds fast; heavy work (email, stock update) happens in background

3. High Throughput
   Handles millions of messages/sec — built for scale

4. Durability
   Messages persisted on disk, replayable — no data loss even if consumer is down

5. Multiple Consumers
   Same event can be consumed by many services (Inventory, Notification, Analytics) 
   without producer changes

6. Fault Tolerance
   Consumer down? Messages wait in Kafka topic until it's back up

7. Ordering Guarantee
   Messages in same partition processed in order (important for order status flow)
