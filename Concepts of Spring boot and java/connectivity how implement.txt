FILEBEAT - WHAT IT IS AND HOW TO USE IT
============================================

WHAT IS FILEBEAT
----------------------------------------------------------------
Filebeat is a lightweight "log shipper" agent. Instead of your
Spring Boot app sending logs directly over TCP to Logstash
(like we did with logstash-logback-encoder), Filebeat sits on
the server, WATCHES a log file, and forwards new lines to
Logstash or Elasticsearch as they are written.

Think of it like: Spring Boot writes logs to a normal .log file
on disk -> Filebeat tails that file -> pushes each new line to
Logstash/Elasticsearch.


WHY USE FILEBEAT INSTEAD OF DIRECT TCP APPENDER
----------------------------------------------------------------
- App doesn't need network dependency on Logstash to log
  (if Logstash is down, app still writes to file safely,
  Filebeat resends later)
- Filebeat is very lightweight (low CPU/RAM) compared to
  running Logstash on every server
- Standard practice in real production/microservices setup
- Works well when you have MANY servers - install Filebeat on
  each, all forward to one central Logstash/Elasticsearch
- Has built-in backpressure handling and retry if Logstash/ES
  is temporarily down


FILEBEAT vs DIRECT TCP APPENDER (what we set up earlier)
----------------------------------------------------------------
DIRECT TCP APPENDER (logstash-logback-encoder):
  Spring Boot app --(TCP, live)--> Logstash --> Elasticsearch
  - Simple, fewer moving parts
  - If Logstash is down when app logs, that log line is LOST
  - Good for single app / dev / small setup

FILEBEAT:
  Spring Boot app --> log file --> Filebeat --> Logstash -->
  Elasticsearch
  - One extra component (Filebeat) but safer, no data loss
  - Good for production, multiple servers, microservices
  - Industry standard approach

ISSUE IDENTIFY AND  IAM FIX 
Scenario: Unbounded Static Cache (Most Common Interview Answer)

sessionCache is static → lives for the entire application lifetime, tied to the class, not GC'd with request scope.
Entries are added on every login but never removed on logout/timeout.
Over days/weeks, heap keeps growing → eventually OutOfMemoryError.


Open VisualVM → Monitor tab → heap graph keeps climbing even after GC (sawtooth trending upward).
Take two heap dumps (early vs after load) → compare in VisualVM.
Sort by instance count → see HashMap$Node and User objects growing linearly with traffic, never dropping.
Right-click User instances → Show Nearest GC Root → traces back to UserSessionService.sessionCache (a static field) → confirms it's the root holding



Scenario 3: Unclosed Resources (Connection/InputStream)
❌ Leaky Code:
javapublic void readData(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path);
    // process file
    // forgot fis.close() — leaks file handle + native memory
}
Leak: Stream/connection never closed → native/heap memory piles up with every call → eventually OutOfMemoryError or "too many open files."



 Fix (try-with-resources):
javapublic void readData(String path) throws IOException {
    try (FileInputStream fis = new FileInputStream(path)) {
        // process file
    } // auto-closed, even on exception
}
One-liner: "Unclosed InputStream/Connection — detected via rising thread/file-handle count in VisualVM, fixed using try-with-resources for guaranteed cleanup."






Public IP vs VPC — Short Explanation
Public IP

An IP address accessible from the internet (anyone, anywhere can reach it)
Used when external users/clients need to connect (e.g., your website, public API)
Risk: exposed to internet attacks if not secured properly

VPC (Virtual Private Cloud)

A private, isolated network inside cloud (AWS/Azure) where your servers talk to each other internally — not exposed to the internet
Services inside VPC use Private IPs to communicate with each other securely


Real-Time Use Case (Why Use Both Differently)
Example: E-commerce app on AWS
Internet Users 
     ↓ (Public IP)
API Gateway / Load Balancer  ← ONLY this is public-facing
     ↓ (Private IP, inside VPC)
Order Service  ←→  Payment Service  ←→  Inventory Service  ←→  Zipkin Server  ←→  MySQL DB

API Gateway: gets a Public IP — users need to reach it from the internet to place orders
Order Service, Payment Service, Inventory Service, Database, Zipkin: stay inside VPC with Private IPs only — no direct internet access, safer from attacks

Why: If Payment Service or your DB had a public IP, hackers could directly attack it. By keeping them inside VPC (private), only your API Gateway is exposed — internal services talk to each other securely without internet exposure.

One-liner to say in interview: "I use a Public IP only for the entry point, like an API Gateway or Load Balancer, so external users can reach the app. All internal microservices, databases, and tools like Zipkin stay inside a VPC using private IPs — so they can communicate with each other securely without being exposed to the internet,

AWS CLOUDWATCH WITH SPRING BOOT MICROSERVICES - COMPLETE GUIDE
==================================================================

WHAT IS CLOUDWATCH?
--------------------
AWS CloudWatch is a monitoring and observability service used to:
- Collect and track LOGS
- Collect and track METRICS
- Set ALARMS on thresholds
- Visualize data using DASHBOARDS
- Trigger automated actions (via EventBridge/Alarms)

==================================================================
WAYS / FEATURES USED WITH SPRING BOOT MICROSERVICES
==================================================================

1. CLOUDWATCH LOGS
   - Centralized logging for all microservices
   - Replaces scattered log files across EC2/ECS/EKS instances

2. CLOUDWATCH METRICS
   - CPU, Memory, JVM heap, thread count, request latency, error rate
   - Custom business metrics (orders processed, payment failures, etc.)

3. CLOUDWATCH ALARMS
   - Auto-notify via SNS/Email/Slack when thresholds breach
   - Example: CPU > 80% for 5 mins -> trigger alarm

4. CLOUDWATCH DASHBOARDS
   - Visual graphs for logs/metrics in one place
   - Used by DevOps/Support teams for real-time monitoring

5. CLOUDWATCH LOGS INSIGHTS
   - Query language to search/filter logs quickly
   - Similar to SQL, used for debugging production issues

6. CLOUDWATCH AGENT
   - Installed on EC2 to push OS-level + application logs to CloudWatch

7. CLOUDWATCH EVENTS / EVENTBRIDGE
   - Trigger Lambda/SNS based on log patterns or scheduled events

MONITORING & PROFILING TOOLS - LOCAL vs PRODUCTION
(Spring Boot Microservices, No Docker)
=========================================================

LOCAL DEVELOPMENT
---------------------------------------------------------
VisualVM          -> USE     - Free, zero-setup, attach directly to local JVM PID. Best for heap/thread/GC debugging while coding.

JMeter             -> USE     - Run quick load tests against localhost:8080 to catch obvious bottlenecks before pushing code.

Prometheus+Grafana  -> USE (lightweight) - Run standalone JARs locally to verify /actuator/prometheus metrics and custom Micrometer counters before deploying.

Zipkin              -> OPTIONAL - Only needed locally if testing multi-service call chains on your machine. Skip if running only one service at a time.

ELK                 -> SKIP    - Too heavy for a laptop (Elasticsearch needs 2-4GB+ RAM). Use console/file logs locally instead.

CloudWatch          -> SKIP    - AWS-only; irrelevant until deployed.

JProfiler           -> SKIP    - Paid, heavyweight. Save for diagnosing a specific hard-to-reproduce production issue, not daily local dev.

LOCAL SUMMARY: VisualVM + JMeter + (optionally) local Prometheus/Grafana. That's it.


PRODUCTION (EC2, No Docker)
---------------------------------------------------------
Prometheus+Grafana  -> USE (PRIMARY) - Main metrics/dashboard/alerting stack. Run as systemd services on a dedicated monitoring EC2 instance.

ELK                 -> USE (PRIMARY) - Centralized logs across all microservice instances. Essential once you have more than 1-2 services (can't SSH into every box to tail -f logs).

Zipkin              -> USE     - Critical for tracing a single request across 5+ services to find latency/error sources. Sample at 10%, not 100%, to reduce overhead.

CloudWatch          -> USE (SECONDARY/BACKUP) - Since already on EC2, use as safety net for infra-level alarms (CPU, disk, auto-scaling triggers). Complements ELK/Prometheus, doesn't replace them.

JProfiler           -> USE (SITUATIONAL) - Attach remotely via SSH tunnel (never expose port publicly) only when diagnosing a specific memory leak or CPU spike. Not left running always-on.

VisualVM            -> RARE    - Can be used similarly via remote JMX for quick checks, but JProfiler is the better production-grade choice if available.

JMeter              -> USE     - Run from a SEPARATE EC2 instance (not production app servers) to load-test staging/production before major releases.

PRODUCTION SUMMARY: Prometheus+Grafana + ELK + Zipkin running always-on, CloudWatch as backup layer, JProfiler/JMeter used on-demand.

centralized logging, and Zipkin for distributed tracing - all as

SCENARIO 1: Diagnosing App Freeze / Hang in Production
------------------------------------------------------
Reason:
Your Spring Boot app on EC2 suddenly stops responding to 
requests — no crash, no error in logs, just hangs. CPU 
usage looks normal, but users get timeouts.

You hit /actuator/threaddump to capture a snapshot of ALL 
threads at that moment. You look for threads stuck in 
"BLOCKED" or "WAITING" state for a long time — e.g. 
multiple threads waiting on the same lock (like a 
synchronized method or a database connection). This tells 
you EXACTLY where the app is stuck, instead of restarting 
the server blindly and losing the chance to find the root 
cause.


SCENARIO 2: Detecting Deadlock Between Two Shared Resources
------------------------------------------------------
Reason:
Your app has a batch job that updates both an in-memory 
cache and a database record, and occasionally two 
requests hang forever together, both waiting on each 
other.

You take a threaddump and search for "Found Java-level 
deadlock" in the output. It clearly lists Thread-A waiting 
to lock Resource-2 (held by Thread-B), and Thread-B waiting 
to lock Resource-1 (held by Thread-A). This gives you 
concrete proof of a deadlock and the exact code path 
(class + line number) causing it, so you can fix lock 
ordering in your code instead of guessing.


SCENARIO 3: High CPU Usage Investigation on EC2 Instance
------------------------------------------------------
Reason:
Your EC2 instance shows 95% CPU usage during peak hours, 
but you don't know which part of the app is causing it — 
could be a bad loop, excessive logging, or inefficient 
JSON parsing.

You take a threaddump, then cross-check it against the 
Linux "top -H" output (which shows CPU usage per OS 
thread). By matching the thread ID (nid) from the 
threaddump to the high-CPU OS thread ID, you find the 
EXACT Java thread and its stack trace consuming CPU — 
e.g. a thread stuck in an infinite retry loop calling an 
external API — so you can fix that specific code, instead 
of just scaling up the EC2 instance size as a workaround.

SCENARIO 1: Investigating Memory Leak Before OutOfMemoryError
------------------------------------------------------
Reason:
Your Spring Boot app on EC2 has been running fine for days, 
but memory usage keeps climbing slowly (checked via 
/actuator/metrics/jvm.memory.used) and never comes back 
down even after GC runs. You suspect a memory leak but 
don't know which object is piling up.

You trigger /actuator/heapdump to download a .hprof file, 
then open it in a tool like Eclipse MAT (Memory Analyzer 
Tool) on your local machine. MAT shows you the "Dominator 
Tree" — e.g. a List inside a Singleton Service that keeps 
growing because entries are added but never removed. This 
gives you the EXACT class and field holding onto memory, 
so you can fix the code, instead of just restarting the 
app every few days as a temporary fix.


SCENARIO 2: Finding Root Cause of Recurring OutOfMemoryError Crashes
------------------------------------------------------
Reason:
Your app crashes with "java.lang.OutOfMemoryError: Java 
heap space" every 2-3 days in production, and by the time 
you check logs after a crash, the JVM has already 
restarted (via systemd) and lost the state.

You configure the JVM startup flag 
-XX:+HeapDumpOnOutOfMemoryError so a heap dump is 
AUTOMATICALLY saved to disk at the moment of the crash 
(instead of relying on manually hitting the endpoint 
before it's too late). You then analyze that dump to see 
what object was consuming the most memory right before the 
crash — e.g. a huge List<byte[]> from unclosed file 
uploads — giving you the real cause instead of just seeing 
"OutOfMemoryError" in the log with no context.


SCENARIO 3: Comparing Before/After Heap Dumps to Confirm a Fix
------------------------------------------------------
Reason:
You suspect a caching layer (e.g. a custom in-memory Map 
used for storing session data) is not evicting old 
entries properly, but you're not 100% sure, and you don't 
want to push a "fix" to production without proof.

You take one heapdump BEFORE your code fix (showing, say, 
50,000 entries in the cache Map after a load test) and 
another heapdump AFTER applying your fix (with eviction 
logic corrected), running the same load test again. 
Comparing both dumps in Eclipse MAT shows the entry count 
now stays capped (e.g. at 1,000), giving you concrete 
evidence the fix works, instead of just hoping it's fixed 
and finding out later in production.

## CloudWatch Trigger → Alarm → Auto Scaling (Short Version)

### IEX Logistics — Peak Hour Dispatch Spike

**Scenario:** Every day 9-11 AM, order dispatch volume spikes as businesses assign drivers.

1. CloudWatch Agent tracks CPU on `dispatch-service` EC2 instances.
2. **Alarm:** CPU > 70% for 3 minutes → ALARM state.
3. **Trigger:** Alarm fires → calls Auto Scaling policy.
4. **Auto Scaling:** ASG adds 2 new EC2 instances, they join the ALB, traffic spreads out, CPU drops back to normal.
5. Once traffic settles (CPU < 30% for 5 min), scale-in removes the extra instances.

**One-liner:** *"CloudWatch monitored CPU on dispatch-service; when it crossed 70% during peak hours, an alarm triggered Auto Scaling to add instances, and scaled back down once traffic normalized."*

@RestController
public class OrderController {

    private final MeterRegistry meterRegistry;

    public OrderController(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
    }

    @GetMapping("/order/{id}")
    public ResponseEntity<Order> getOrder(@PathVariable String id) {
        Timer.Sample sample = Timer.start(meterRegistry);
        try {
            Order order = orderService.findById(id);
            return ResponseEntity.ok(order);

            meterRegistry.counter("order.fetch.errors", "reason", e.getClass().getSimpleName()).increment();

        }
    }

@RestController
public class AuthController {

    private final Counter loginSuccessCounter;
    private final Counter loginFailureCounter;

    public AuthController(MeterRegistry registry) {
        this.loginSuccessCounter = Counter.builder("login.attempts")
                .tag("result", "success")
                .register(registry);

        this.loginFailureCounter = Counter.builder("login.attempts")
                .tag("result", "failure")
                .register(registry);
    }

    @PostMapping("/login")
    public ResponseEntity<String> login(@RequestBody LoginRequest req) {
        boolean valid = authService.validate(req);
        if (valid) {
            loginSuccessCounter.increment();
            return ResponseEntity.ok("Login success");
        } else {
            loginFailureCounter.increment();
            return ResponseEntity.status(401).body("Login failed");
        }
    }
}

Prometheus Query:
rate(login_attempts_total{result="failure"}[5m])

Alert if failure rate > 10/min

WHY KAFKA + WEBSOCKET (YOUR SETUP: BIDDING SERVICE + WEBSOCKET SERVICE + DRIVER SERVICE WITH 2 INSTANCES)
YOUR SERVICES

Bidding Service       -> creates order/bid, decides nearby drivers
WebSocket Service     -> holds live socket connections with driver apps
Driver Service        -> 2 instances (load balanced) -> updates driver location/status

PROBLEM WITHOUT KAFKA
Bidding Service wants to notify a driver.
But that driver's live WebSocket connection is sitting inside "WebSocket Service" -
a completely different service, running as its own process.
So Bidding Service cannot directly touch that connection.
It would need to call WebSocket Service over REST/HTTP just to say
"push this to driver X" - this creates tight coupling:
Bidding Service --> HTTP call --> WebSocket Service --> push to driver
Problems with this direct HTTP approach:

If WebSocket Service is restarting/deploying/crashed for a few seconds,
the HTTP call fails and the event is LOST (no retry, no memory of it)
Bidding Service now has to know WebSocket Service's URL, retry logic, timeouts
If tomorrow you add Notification Service / Analytics Service / Audit Service,
Bidding Service needs to call ALL of them one by one -> more coupling, more risk

WHY KAFKA FIXES THIS
Bidding Service just publishes event to Kafka topic "order-events" and forgets about it.
It does NOT call WebSocket Service directly. No knowledge of WebSocket Service's
existence, address, or state.
Bidding Service --> Kafka topic (order-events)
WebSocket Service --> subscribes to topic --> pushes to driver
Benefits in YOUR exact setup:

DECOUPLING
Bidding Service and WebSocket Service don't call each other directly.
Bidding Service doesn't care who or how many services read the event.
NO EVENT LOSS
If WebSocket Service is down/restarting for a few seconds,
Kafka just holds the event in the topic.
When WebSocket Service comes back, it reads from where it left off.
With direct HTTP call, that event would just be gone forever.
DRIVER SERVICE HAS 2 INSTANCES - SAME PROBLEM EXISTS THERE TOO
If Driver Service (2 instances) needs to react to order events too
(say, to update driver's current job status), both instances can
independently consume from Kafka without Bidding Service needing to
know there are 2 instances, or manage load balancing itself.
Kafka consumer groups handle this automatically - only one instance
processes a given event, no duplicate processing.
EASY TO ADD MORE CONSUMERS LATER
Tomorrow if you add Notification Service (SMS/push) or Audit Service,
they just subscribe to the same Kafka topic.
Bidding Service code does NOT change at all.
HANDLES TRAFFIC BURSTS
If many bids/orders happen at once, Kafka queues them up.
WebSocket Service consumes at its own pace instead of getting
overwhelmed by a sudden spike of direct calls.

WHEN WEBSOCKET ALONE (NO KAFKA) WOULD BE OK

If Bidding Service and WebSocket Service were the SAME service/process
If you only ever have ONE instance of everything
If losing an occasional event during a restart is acceptable
Early prototype / very low traffic

SIMPLE ONE LINE ANSWER
WebSocket = HOW the message reaches the driver's live connection.
Kafka = HOW the message reliably GETS FROM Bidding Service TO WebSocket Service
(and any other service) without them being directly wired to each other,
without losing events on restart, and without breaking when you scale
instances or add new services.
In your case, since Bidding Service, WebSocket Service, and Driver Service are
3 SEPARATE services (not one app) - Kafka is justified, not overkill.

















==========================================================
KAFKA CONSUMER LAG - REASONS & SOLUTIONS
==========================================================

WHAT IS CONSUMER LAG:
Lag = Producer's latest offset - Consumer's current offset
= Number of messages published but not yet processed by consumer


----------------------------------------------------------
1. CONSUMER IS SLOW (PROCESSING BOTTLENECK)
----------------------------------------------------------
REASON:
Listener code does heavy work per message - DB calls, WebSocket 
push, external API calls - each message takes too long.

Example:
@KafkaListener(topics = "order-events")
public void onOrderEvent(OrderEvent event) {
    driverRepository.findById(event.getDriverId()); // slow DB call
    simpMessagingTemplate.convertAndSend(...);       // network call
}

SOLUTION:
- Keep listener lightweight - only essential work (push to WebSocket)
- Move heavy work (logging, analytics) to async/separate service
- Cache driver lookups instead of DB call every time
- Use @Async or push slow tasks to another Kafka topic


----------------------------------------------------------
2. NOT ENOUGH CONSUMER INSTANCES (PARALLELISM ISSUE)
----------------------------------------------------------
REASON:
A topic has multiple partitions. Only one consumer per partition
can read at a time within a group. Too few instances = bottleneck.

Example:
Topic: order-events (6 partitions)
Consumer Group: driver-notify-group
Only 1 instance running = 1 consumer handling all 6 partitions = SLOW

SOLUTION:
- Scale consumer instances to match partition count
- Deploy multiple instances of the app with same group-id
  Kafka auto-balances partitions across them

application.yml:
spring:
  kafka:
    consumer:
      group-id: driver-notify-group


----------------------------------------------------------
3. SUDDEN TRAFFIC SPIKE (PEAK HOURS)
----------------------------------------------------------
REASON:
Lunch rush, festival sale, flash promo - orders flood in faster
than consumer can keep up, even if normally fine.

SOLUTION:
- Auto-scale consumer instances based on lag metric
  (Kubernetes HPA with custom Kafka lag metric)
- Increase partition count in advance of expected high traffic
  (Note: partitions can only be increased, not decreased)














----------------------------------------------------------
4. CONSUMER CRASHED OR RESTARTING
----------------------------------------------------------
REASON:
Service crashed (OOM, deployment restart, pod eviction) - no one
was consuming for a few minutes, messages piled up.

SOLUTION:
- Set up health checks / auto-restart (K8s liveness probes,
  or systemd restart policy if not using K8s)
- Alert immediately on lag spike to catch crashes fast
- Fix root cause - check memory limits, heap size (-Xmx), GC logs


----------------------------------------------------------
5. REBALANCING STORMS
----------------------------------------------------------
REASON:
Every time a consumer joins/leaves the group (deploys, crashes,
scales), Kafka triggers a rebalance - pauses ALL consumers briefly
to reassign partitions. Frequent restarts = frequent pauses = lag builds.

SOLUTION:
- Avoid frequent restarts/deployments during peak hours
- Tune session.timeout.ms and max.poll.interval.ms so Kafka
  doesn't falsely mark a slow-but-alive consumer as dead
- Use Cooperative Sticky Assignor instead of default range assignor

application.yml:
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor


----------------------------------------------------------
6. POISON PILL MESSAGE (STUCK MESSAGE)
----------------------------------------------------------
REASON:
One bad message (malformed JSON, null field causing exception)
keeps failing and retrying forever, blocking everything behind
it on that partition.

SOLUTION:
Add a Dead Letter Topic (DLT) - bad messages get moved aside
instead of blocking the queue.

Code:
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
    DeadLetterPublishingRecoverer recoverer =
        new DeadLetterPublishingRecoverer(template);
    return new DefaultErrorHandler(recoverer, new FixedBackOff(1000L, 3));
    // retry 3 times then move to DLT
}


----------------------------------------------------------
7. POOR BATCH/FETCH CONFIG
----------------------------------------------------------
REASON:
Consumer is fetching too few records per poll, wasting round trips.

SOLUTION:
Tune fetch settings:

application.yml:
spring:
  kafka:
    consumer:
      max-poll-records: 500      # process more per poll
      fetch-min-size: 1048576    # 1MB minimum fetch batch

                                  ^           ^

Kafka server / broker	Same thing, two names. One running Kafka process that stores data and handles requests.
Kafka cluster	A group of multiple brokers working together as one system.
Picture it like this
Kafka cluster
 ├── Broker 1 (a Kafka server running on Machine 1)
 ├── Broker 2 (a Kafka server running on Machine 2)
 └── Broker 3 (a Kafka server running on Machine 3)

"Broker" and "Kafka server" are literally the same thing — just two names people use interchangeably for one running instance of Kafka.

"Cluster" is the group of them together.

Your setup right now

You've been running 1 broker = 1 server = technically a cluster of size 1. That's why replication factor had to be 1 — there was only one broker to put data on, nothing else to copy it to.

One line

Broker = server = one Kafka process. Cluster = several brokers teamed up. Everything else we discussed — partitions, replicas, leader election — only becomes meaningful once you have more than one broker in the cluster.


Partition	Leader (handles all reads/writes)	Replica copies (standby, kept in sync)
Partition 0	Broker 1	Broker 2, Broker 3
Partition 1	Broker 2	Broker 1, Broker 3
Partition 2	Broker 3	Broker 1, Broker 2





What happens if two consumers in the same group try to read the same partition?

It can't happen — Kafka blocks it by design. One partition can only be assigned to one consumer within a group at a time. If a second consumer tries to grab it, a rebalance occurs and Kafka reassigns cleanly — there's never a moment where two consumers in the same group are actively reading the same partition simultaneously.


What happens if you have more consumers than partitions?

The extra consumers get zero partitions assigned — they sit idle, doing nothing, just members of the group. They only become active if an existing consumer crashes and a partition needs reassigning. (Example: 3 partitions, 5 consumers → 2 consumers are pure idle standby.)



What happens if a producer sends a message but never gets an acknowledgment — did it actually get delivered?

The producer sends a message. The broker saves it and sends back "success" — but that success reply gets lost in the network.

The producer never receives the confirmation, so it can't tell if the message was saved or not. To be safe, it resends the same message.

Problem: if the original message did save successfully, you now have it saved twice — a duplicate.

Fix: enable.idempotence: true — Kafka tags each message with a unique sequence number. If the same numbered message arrives twice, the broker recognizes it and ignores the duplicate, keeping only one copy.

7. DISTRIBUTED LOCK (MANUAL - setIfAbsent)
==========================================================

EXAMPLE 1: Prevent Double Order Assignment
-----------------------------------------------
@Service
public class OrderAssignmentService {

    private final RedisTemplate<String, String> redisTemplate;

    public OrderAssignmentService(RedisTemplate<String, String> redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    public boolean assignOrder(String orderId, String driverId) {
        String lockKey = "lock:order:" + orderId;
        Boolean lockAcquired = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, driverId, Duration.ofSeconds(10));

        if (Boolean.TRUE.equals(lockAcquired)) {
            try {
                orderRepository.assignDriver(orderId, driverId);
                return true;
            } finally {
                redisTemplate.delete(lockKey);
            }
        }
        return false;
    }
}
-> Two dispatch requests race to assign same order -> only one wins.

==========================================================
READ-THROUGH CACHE - CODE EXAMPLES (SPRING BOOT + REDIS)
==========================================================

CONCEPT:
--------
Cache-Aside: YOUR code checks cache, fetches DB on miss, stores in cache.
Read-Through: THE CACHE LAYER itself owns "check -> miss -> fetch -> store".
Spring's @Cacheable gives you read-through BEHAVIOR via AOP proxy.


----------------------------------------------------------
VERSION 1: @Cacheable (SPRING-MANAGED READ-THROUGH)
----------------------------------------------------------

@Service
@CacheConfig(cacheNames = "orders")
public class OrderService {

    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @Cacheable(key = "#orderId")
    public Order getOrderById(String orderId) {
        // Spring's proxy wraps this method:
        // 1. Intercepts the call
        // 2. Checks Redis for key "orders::orderId"
        // 3. HIT -> returns cached value, method body never runs
        // 4. MISS -> runs this method, stores result in Redis, returns it
        System.out.println("DB HIT - fetching from database: " + orderId);
        return orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));
    }
}

NOTE: No manual Redis code here (no redisTemplate.get/set).
Spring's AOP proxy handles it - this is the "framework-managed" part.


----------------------------------------------------------
VERSION 2: MANUAL READ-THROUGH (BUILD YOUR OWN LOADER PATTERN)
----------------------------------------------------------

Generic reusable cache class:
-------------------------------
@Component
public class ReadThroughCache<K, V> {

    private final RedisTemplate<String, Object> redisTemplate;
    private final String cachePrefix;
    private final Duration ttl;

    public ReadThroughCache(RedisTemplate<String, Object> redisTemplate, String cachePrefix, Duration ttl) {
        this.redisTemplate = redisTemplate;
        this.cachePrefix = cachePrefix;
        this.ttl = ttl;
    }

    // "loader" = the DB-fetch function passed in by caller
    public V get(K key, Function<K, V> loader) {
        String redisKey = cachePrefix + ":" + key;

        // Step 1: Check cache
        V cached = (V) redisTemplate.opsForValue().get(redisKey);
        if (cached != null) {
            System.out.println("CACHE HIT: " + redisKey);
            return cached;
        }

        // Step 2: Miss -> cache layer calls the loader (DB fetch)
        System.out.println("CACHE MISS - loading from source: " + redisKey);
        V value = loader.apply(key);

        // Step 3: Store result back into cache before returning
        if (value != null) {
            redisTemplate.opsForValue().set(redisKey, value, ttl);
        }

        return value;
    }
}


Usage in service:
--------------------
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final ReadThroughCache<String, Order> orderCache;

    public OrderService(OrderRepository orderRepository, RedisTemplate<String, Object> redisTemplate) {
        this.orderRepository = orderRepository;
        this.orderCache = new ReadThroughCache<>(redisTemplate, "orders", Duration.ofMinutes(10));
    }

    public Order getOrderById(String orderId) {
        return orderCache.get(orderId, id ->
                orderRepository.findById(id)
                        .orElseThrow(() -> new OrderNotFoundException(id))
        );
    }
}

COMPARISON TABLE
----------------------------------------------------------
ASPECT                        | @Cacheable (Spring)         | Manual ReadThroughCache
-------------------------------|------------------------------|---------------------------
Who checks Redis first        | Spring AOP proxy (hidden)   | ReadThroughCache.get()
Who calls DB on miss           | Spring proxy invokes method | You pass loader function
Who writes result to Redis     | Spring proxy (after return) | ReadThroughCache.get()
Your code writes                | Just DB fetch logic         | Just DB fetch as lambda
Works outside Spring beans      | No (needs AOP proxy)        | Yes (plain Java, reusable)

BOTTOM LINE:
Both are read-through in behavior - cache owns fetch-on-miss flow.
@Cacheable = Spring AOP magic.
ReadThroughCache = same pattern built manually, more control
(custom fallback, logging, metrics per miss).


==========================================================
REDIS CACHING STRATEGIES - FULL SUMMARY
==========================================================

1. CACHE-ASIDE (LAZY LOADING)
--------------------------------
Flow: App checks cache -> MISS -> fetch DB -> store in cache -> return
      App checks cache -> HIT -> return (no DB call)

Code:
@Cacheable(value = "orders", key = "#orderId")
public Order getOrderById(String orderId) {
    return orderRepository.findById(orderId).orElseThrow();
}

Pros: Simple, only caches what's requested
Cons: First request always slow, can go stale


2. WRITE-THROUGH
--------------------
Flow: App writes to Redis -> Redis writes to DB -> Confirm (synchronous)

Code:
@CachePut(value = "orders", key = "#order.id")
public Order updateOrder(Order order) {
    return orderRepository.save(order);
}

Pros: Cache always in sync with DB
Cons: Writes slightly slower (2 operations)


3. WRITE-BEHIND (WRITE-BACK)
--------------------------------
Flow: App writes to Redis (fast) -> background job batches to DB later

Code:
@Service
public class DriverLocationService {

    private final RedisTemplate<String, String> redisTemplate;

    public void updateLocation(String driverId, double lat, double lng) {
        redisTemplate.opsForValue().set("driver:location:" + driverId, lat + "," + lng);
    }

    @Scheduled(fixedRate = 60000)
    public void persistLocationsToDb() {
        Set<String> keys = redisTemplate.keys("driver:location:*");
        for (String key : keys) {
            String value = redisTemplate.opsForValue().get(key);
            // parse and bulk save to DB
        }
    }
}

Pros: Very fast writes, good for high-frequency data (GPS)
Cons: Risk of data loss if Redis crashes before DB sync


4. READ-THROUGH
--------------------
Flow: Cache layer itself owns "check -> miss -> fetch -> store"
(see full code examples above)

Pros: Clean separation, reusable, framework or custom-managed
Cons: Slightly more setup than plain cache-aside


5. TTL-BASED EXPIRY
------------------------
Flow: Data auto-expires after set duration, no manual invalidation

Code:
redisTemplate.opsForValue().set("otp:" + phoneNumber, otp, Duration.ofMinutes(2));
redisTemplate.opsForValue().set("driver:location:" + driverId, json, Duration.ofMinutes(5));

Use for: OTP, live location, session tokens, rate-limit counters


6. EVICT-ON-WRITE (CACHE INVALIDATION)
-------------------------------------------
Flow: On write, delete cache entry instead of updating it.
Next read repopulates fresh from DB.

Code:
@CacheEvict(value = "orders", key = "#orderId")
public void updateOrderStatus(String orderId, String status) {
    orderRepository.updateStatus(orderId, status);
}

Pros: Simpler than write-through
Cons: Next read after write is a cache miss (slightly slower once)


----------------------------------------------------------
QUICK COMPARISON TABLE
----------------------------------------------------------
STRATEGY        | WRITE SPEED   | READ SPEED         | FRESHNESS      | BEST FOR
-----------------|----------------|---------------------|-----------------|---------------------------
Cache-Aside      | Normal         | Fast after 1st hit  | Can go stale   | General lookup (orders)
Write-Through    | Slower         | Always fast          | Always fresh   | Critical consistent data
Write-Behind     | Very fast      | Fast                 | Risk of loss   | High-frequency writes (GPS)
Read-Through     | N/A (read op)  | Fast after 1st hit   | Depends on TTL | Clean abstraction, reusable
TTL Expiry        | Normal         | Fast until expiry    | Self-cleaning  | OTP, sessions, rate limits
Evict-on-Write   | Fast           | Fast (miss on 1st)   | Fresh (lazy)   | Simple write alternative


----------------------------------------------------------
WHICH TO USE IN LOGISTICS APP
----------------------------------------------------------
Order lookup (driver checks status)     -> Cache-Aside / Read-Through
Order status update (dispatcher edits)  -> Evict-on-Write or Write-Through
Live GPS location (every few sec)       -> Write-Behind + TTL
OTP / login session                     -> TTL Expiry
Rate limiting API calls                 -> TTL Expiry (setIfAbsent)

==========================================================
REDIS TIERS - INTERVIEW QUICK ANSWERS
==========================================================

1. STANDALONE
   One Redis server. No HA. Crash = downtime.

2. REPLICATION (Master-Replica)
   Master handles writes, replicas hold read-only copies.
   Async replication (near real-time, tiny lag possible).
   NO auto-failover if master dies.

3. SENTINEL
   3+ Sentinel processes monitor master via heartbeat.
   Master down -> Sentinels vote (quorum) -> promote best
   replica -> app auto-reconnects to new master.
   Use for: production HA, single dataset.

4. CLUSTER
   Data sharded across N masters using 16384 hash slots.
   Each master has its own replica(s) for failover.
   Client connects to any node, gets redirected to correct one.
   Use for: large datasets, high write throughput, horizontal scale.


==========================================================
SYNC MECHANISM (LIKELY FOLLOW-UP QUESTION)
==========================================================
Initial sync: Master runs BGSAVE -> sends RDB snapshot to replica
Ongoing sync: Every write on master is streamed to replica
Type: Asynchronous by default (master doesn't wait for replica ack)
Tradeoff: Fast, but a write can be lost if master crashes before
          replicating it (use WAIT command for stronger guarantee)


==========================================================
FAILOVER MECHANISM (LIKELY FOLLOW-UP QUESTION)
==========================================================
Sentinel pings master every second
  -> Master unresponsive -> marked SDOWN (subjectively down)
  -> Other Sentinels confirm -> ODOWN (objectively down) via quorum
  -> Sentinels elect a leader among themselves
  -> Leader picks best replica (least lag) and promotes it
  -> Other replicas re-point to new master
  -> Sentinels notify app of new master address

KEY POINT: Replica does NOT decide on its own. Promotion is a
voting process among Sentinels (or cluster nodes), not automatic
self-promotion. This avoids false failover from network blips.


==========================================================
ONE-LINE ANSWER IF ASKED DIRECTLY
==========================================================
"Master streams writes to replica asynchronously. If master
fails, it's not the replica that decides to become master -
Sentinels (or cluster nodes) detect the failure via quorum
voting, then promote the most up-to-date replica and redirect
all clients to it automatically."
