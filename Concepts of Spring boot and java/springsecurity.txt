dofilter:

Think of doFilter as the green light that tells a request, "You are cleared to move to the next step."

Inside a security filter, filterChain.doFilter(request, response) literally means: "I am done checking this request. Pass it along to the next bouncer in line, or let it finally reach the Controller."


Without the call: The filter chain breaks, halting execution, and the client receives an empty response.

 BCryptPasswordEncoder:
What it is: A one-way hashing algorithm to store passwords securely in the database.
One-way — you cannot reverse a BCrypt hash back to the original password
Salted automatically — same password → different hash every time (prevents rainbow table attacks)
Slow by design — deliberately CPU-intensive (adaptive cost factor) to resist brute-force attacks

Spring never decrypts the stored hash — it re-hashes the input and compares hashes.




The Core Problem in Microservices

"In a monolith, one app does login + validation. In microservices, you have an Auth Service issuing tokens, and multiple other services (Order, Payment, Inventory) that need to validate tokens — but shouldn't all share the same secret key or call Auth Service on every request."

Signature Key: Symmetric vs Asymmetric (the big shift)
❌ Symmetric (HMAC) — bad fit for microservices
javaKeys.hmacShaKeyFor(SECRET.getBytes());
Problem: Every microservice needs the same secret to verify tokens → if you distribute the secret to 10 services, and one gets compromised, attacker can forge tokens for the entire system.


✅ Asymmetric (RSA/EC) — correct approach

Interview one-liner:

"In microservices, we sign with RS256 (private key) at the Auth Service and distribute only the public key to other services for verification — this way, no service other than Auth can forge a token, even if compromised."

"If services just trust X-User-Id header, what stops someone from bypassing the Gateway and calling Order Service directly with a fake header?"


Solution 2: mTLS (mutual TLS) between Gateway and services
Gateway ──[client cert]──► Order Service ──[verifies Gateway's cert]──► accepts request

Each service verifies the caller's certificate, not just trusts any request blindly
Even if someone bypasses network isolation, they can't forge a valid mTLS handshake without the Gateway's private key




Secret Key
A password known only to your server, used to sign and verify tokens.
javaString SECRET = "my-super-secret-key-12345";
Think of it like a stamp only your server owns. Nobody else has this stamp.

Signature
The proof that a token is genuine and untouched — created by combining the token's data with the secret key.
Signature = Hash(Header + Payload, SECRET_KEY)
JWT looks like this:
eyJhbGc...  .  eyJzdWI...  .  SflKxwRJ...
  HEADER          PAYLOAD        SIGNATURE

How it works — simple flow
1. Server creates token (login time)
Data: "username=thiru"
Server stamps it with SECRET_KEY → Signature: "abc123xyz"
Token sent to client: username=thiru + abc123xyz
2. Client sends token back (every request)
Client sends: username=thiru + abc123xyz
3. Server checks (every request)
Server re-calculates: Hash("username=thiru", SECRET_KEY) 
If result == abc123xyz → ✅ Valid, not tampered
If result != abc123xyz → ❌ Rejected, someone modified it

WHY TWO TOKENS? (the core problem)
-----------------------------------------------------
- Long-lived token = if stolen, attacker has access
  for a long time (dangerous)
- Short-lived token = safer, but forces user to
  login again and again (bad experience)
- Solution: use BOTH → short-lived access token +
  long-lived refresh token



ACCESS TOKEN
-----------------------------------------------------
Purpose   : Used to access protected APIs/resources
Lifespan  : Short (5-15 mins typically)
Sent in   : Authorization: Bearer <token> header,
            on every API request
Contains  : username, roles, expiry
Risk if   : Low - expires quickly, limited damage



REFRESH TOKEN
-----------------------------------------------------
Purpose   : Used ONLY to get a new access token
            (not used to access APIs directly)
Lifespan  : Long (7-30 days, sometimes months)
Sent in   : Only to a dedicated /refresh-token endpoint
Contains  : minimal info, often just a reference/ID
Risk if   : High - long-lived, so must be protected
stolen      carefully (HttpOnly secure cookie)



FLOW
-----------------------------------------------------
1. Login → server issues:
     - Access Token (short life)
     - Refresh Token (long life)

2. Client calls APIs using Access Token

3. Access Token expires (15 min later)

4. Client calls /refresh-token with Refresh Token

5. Server validates refresh token (checks DB/Redis)
   → issues NEW access token (+ optionally new
     refresh token if rotation enabled)

6. Client continues using new access token
   -- user never had to "login" again




WHY THIS DESIGN (real world reasoning)
-----------------------------------------------------
- Access token: stateless, fast, but short-lived
  → limits damage if leaked
- Refresh token: stateful (DB-backed), so it CAN be
  revoked/blacklisted → gives control back to server
  → balances SECURITY + USER EXPERIENCE

Step 3: Minimal Security Configuration ClassCreate this configuration class inside each microservice. It tells Spring Security to validate incoming tokens against the jwk-set-uri specified in your YAML file.Java@Configuration
@EnableWebSecurity
@EnableMethodSecurity // Allows @PreAuthorize on your controllers
public class ResourceServerConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable()) // Stateless APIs don't need CSRF
            .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/v1/public/**").permitAll()
                .anyRequest().authenticated()
            )
            // Tells Spring Boot to act as a Resource Server and expect JWTs
            .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()));

        return http.build();
    }
}

=====================================================
CSRF vs XSS vs CORS - SPRING SECURITY HANDLING
=====================================================

-----------------------------------------------------
1. CSRF (Cross-Site Request Forgery)
-----------------------------------------------------
WHAT IT IS:
- Attacker tricks a LOGGED-IN user's browser into
  sending an unwanted request to your site
  (using the browser's auto-sent session cookie)

EXAMPLE:
- You're logged into bank.com (session cookie active)
- You visit evil.com which has:
  <img src="bank.com/transfer?amount=5000&to=hacker">
- Browser auto-attaches your bank.com cookie
- Transfer happens WITHOUT you knowing

WHO IS VULNERABLE:
- Cookie/session based auth (browser auto-sends cookies)

SPRING SECURITY HANDLING:
- Default: CSRF protection ENABLED for session-based apps
- Spring generates a CSRF token, must be sent back in
  every state-changing request (POST/PUT/DELETE)
- Attacker's page can't read/know this token → blocked

  http.csrf(csrf -> csrf
      .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse()));

FOR JWT (STATELESS) APPS:
- CSRF is usually DISABLED
- WHY: JWT sent in Authorization header manually by
  JS code, NOT auto-attached by browser like cookies
- Attacker's page CANNOT read your JWT from
  localStorage/memory → CSRF attack not possible

  http.csrf(csrf -> csrf.disable());

REAL WORLD RULE:
- Session/cookie auth → CSRF protection ON
- JWT header-based auth → CSRF protection OFF (not needed)

=====================================================

2. XSS (Cross-Site Scripting)
-----------------------------------------------------
WHAT IT IS:
- Attacker injects malicious JavaScript into your
  website (via input fields, comments, URL params)
  which runs in OTHER users' browsers

EXAMPLE:
- Comment box accepts: <script>stealCookie()</script>
- If not sanitized, this script runs for every user
  who views that comment → steals their token/cookie

HOW IT STEALS JWT:
- If JWT stored in localStorage → JS-accessible
  → XSS script can read it: localStorage.getItem("token")
  → sent to attacker's server

SPRING SECURITY / APP-LEVEL HANDLING:
- Spring Security sets security headers by default:

  http.headers(headers -> headers
      .contentSecurityPolicy(csp -> csp
          .policyDirectives("script-src 'self'"))
      .xssProtection(Customizer.withDefaults()));

- X-Content-Type-Options: nosniff
- Content-Security-Policy (CSP) - restricts which
  scripts can run
- Input validation/sanitization at code level
  (escape HTML, use libraries like OWASP Java Encoder)
- Frontend: use frameworks like React/Angular which
  auto-escape output by default

BEST PRACTICE FOR JWT STORAGE (XSS defense):
- DON'T store JWT in localStorage (JS-readable)
- STORE in HttpOnly Secure Cookie (JS CANNOT read it,
  even if XSS script runs)

REAL WORLD RULE:
- Sanitize all user input
- Set CSP headers
- Prefer HttpOnly cookies for tokens over localStorage

=====================================================

3. CORS (Cross-Origin Resource Sharing)
-----------------------------------------------------
WHAT IT IS:
- Browser security rule: JS on domainA.com CANNOT call
  API on domainB.com UNLESS domainB.com explicitly
  allows it
- NOT an attack itself - it's a BROWSER PROTECTION,
  but must be configured correctly or legit frontend
  calls get blocked

EXAMPLE:
- Frontend: myapp.com (React/Angular)
- Backend API: api.myapp.com
- Browser blocks the call unless backend sends
  proper CORS headers allowing myapp.com

SPRING SECURITY HANDLING:

  @Bean
  public CorsConfigurationSource corsConfigurationSource() {
      CorsConfiguration config = new CorsConfiguration();
      config.setAllowedOrigins(List.of("https://myapp.com"));
      config.setAllowedMethods(List.of("GET","POST","PUT","DELETE"));
      config.setAllowedHeaders(List.of("Authorization","Content-Type"));
      config.setAllowCredentials(true);

      UrlBasedCorsConfigurationSource source =
          new UrlBasedCorsConfigurationSource();
      source.registerCorsConfiguration("/**", config);
      return source;
  }

  http.cors(cors -> cors.configurationSource(corsConfigurationSource()));

REAL WORLD RULE:
- NEVER use allowedOrigins("*") in production with
  allowCredentials(true) - security hole
- Whitelist ONLY known frontend domains
- Different CORS config per environment
  (localhost:4200 for dev, real domain for prod)
