Input Validation in Spring Boot: Concepts & Implementation
Core Idea: Input validation ensures that data received from clients (e.g., via REST API requests) conforms to predefined rules and constraints before it's processed by your business logic. This prevents invalid data from corrupting your application, database, or causing security vulnerabilities.

Key Components in Spring Boot:

Spring Boot leverages the Java Bean Validation API (JSR 380, currently Jakarta Validation 3.0, previously Javax Validation 2.0). Hibernate Validator is the reference implementation commonly used.

1. Annotations for Defining Constraints
You apply validation annotations directly to the fields of your DTOs (Data Transfer Objects) or entities.

Dependency:
You need to include the spring-boot-starter-validation dependency.

XML

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
Common Built-in Annotations:

@NotNull: The annotated element must not be null.

@NotEmpty: The annotated element must not be null or empty (for collections, arrays, and strings).

@NotBlank: The annotated string must not be null and must contain at least one non-whitespace character.

@Size(min, max): The annotated element's size must be between min and max (inclusive) for collections, maps, arrays, and strings.

@Min(value): The annotated numeric value must be greater than or equal to value.

@Max(value): The annotated numeric value must be less than or equal to value.

@Positive, @PositiveOrZero, @Negative, @NegativeOrZero: For numeric values.

@Email: The annotated string must be a well-formed email address.

@Pattern(regexp, flags): The annotated string must match the specified regular expression.

@Future, @Past, @FutureOrPresent, @PastOrPresent: For date/time values.

@AssertTrue, @AssertFalse: For boolean values to assert conditions.

Code Example: DTO with Validation Annotations

Java

// Assuming Jakarta Validation (Spring Boot 3.x)
import jakarta.validation.constraints.DecimalMin;
import jakarta.validation.constraints.Email;
import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.NotNull;
import jakarta.validation.constraints.Pattern;
import jakarta.validation.constraints.Size;

// For Spring Boot 2.x, use javax.validation.constraints.*

public class ProductRequestDTO {

    @NotBlank(message = "Product name cannot be empty")
    @Size(min = 3, max = 100, message = "Product name must be between 3 and 100 characters")
    private String name;

    @NotNull(message = "Price is required")
    @DecimalMin(value = "0.01", message = "Price must be positive")
    private Double price;

    @NotBlank(message = "Category cannot be empty")
    @Pattern(regexp = "^(ELECTRONICS|BOOKS|FOOD)$", message = "Category must be ELECTRONICS, BOOKS, or FOOD")
    private String category;

    @Email(message = "Supplier email must be a valid email address")
    @NotBlank(message = "Supplier email is required")
    private String supplierEmail;

    // Constructors, Getters, Setters
    public ProductRequestDTO() {}

    public ProductRequestDTO(String name, Double price, String category, String supplierEmail) {
        this.name = name;
        this.price = price;
        this.category = category;
        this.supplierEmail = supplierEmail;
    }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public Double getPrice() { return price; }
    public void setPrice(Double price) { this.price = price; }
    public String getCategory() { return category; }
    public void setCategory(String category) { this.category = category; }
    public String getSupplierEmail() { return supplierEmail; }
    public void setSupplierEmail(String supplierEmail) { this.supplierEmail = supplierEmail; }
}
2. Triggering Validation in Controllers
To activate the validation defined on your DTO, you need to use specific annotations in your Spring Controller.

@Valid: This is the standard annotation from jakarta.validation (or javax.validation). You typically use it on parameters in your @RequestBody methods. When @Valid is encountered, Spring triggers the validation process.

@Validated: This is a Spring-specific annotation. It's more powerful than @Valid because it supports validation groups and can be used on class-level for method validation (e.g., on a @Service class) or on method parameters where @Valid might not be applicable (e.g., a simple String parameter in a @GetMapping). For @RequestBody, @Valid is sufficient and commonly used.

Code Example: Controller Using @Valid

Java

import jakarta.validation.Valid; // For @Valid annotation
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/products")
public class ProductController {

    // Imagine a ProductService for business logic
    // private final ProductService productService;

    // public ProductController(ProductService productService) {
    //     this.productService = productService;
    // }

    @PostMapping
    public ResponseEntity<String> createProduct(@Valid @RequestBody ProductRequestDTO productDto) {
        // If validation passes, this code will be executed.
        // If validation fails, Spring will throw a MethodArgumentNotValidException
        // which we'll handle globally.
        System.out.println("Product received and validated: " + productDto.getName());
        // productService.saveProduct(productDto); // Call your service layer
        return ResponseEntity.status(HttpStatus.CREATED).body("Product created successfully!");
    }

    @PutMapping("/{id}")
    public ResponseEntity<String> updateProduct(@PathVariable Long id, @Valid @RequestBody ProductRequestDTO productDto) {
        System.out.println("Product update received and validated for ID: " + id);
        // productService.updateProduct(id, productDto);
        return ResponseEntity.ok("Product updated successfully!");
    }
}
3. Handling Validation Errors (Returning Meaningful Messages)
By default, if validation fails, Spring will throw a MethodArgumentNotValidException. Without explicit handling, this would result in a generic 500 Internal Server Error, which isn't user-friendly.

The best practice is to use a @ControllerAdvice to create a global exception handler for validation errors.

Code Example: Global Exception Handler

Java

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.validation.FieldError;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

import java.util.HashMap;
import java.util.Map;

@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, String>> handleValidationExceptions(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach(error -> {
            String fieldName = ((FieldError) error).getField();
            String errorMessage = error.getDefaultMessage();
            errors.put(fieldName, errorMessage);
        });
        return new ResponseEntity<>(errors, HttpStatus.BAD_REQUEST); // HTTP 400
    }

    // You might add other general exception handlers here for 500 errors etc.
    // @ExceptionHandler(Exception.class)
    // public ResponseEntity<String> handleGenericException(Exception ex) {
    //     return new ResponseEntity<>("An unexpected error occurred: " + ex.getMessage(), HttpStatus.INTERNAL_SERVER_ERROR);
    // }
}
Expected Client Response for Invalid Input (HTTP 400 Bad Request):

If a client sends an invalid ProductRequestDTO (e.g., name is empty, price is negative), they would receive something like:

JSON

{
    "name": "Product name cannot be empty",
    "price": "Price must be positive",
    "category": "Category must be ELECTRONICS, BOOKS, or FOOD"
}
This provides clear, field-specific error messages.

4. Custom Constraints (Advanced)
Sometimes, built-in annotations aren't enough. You can create your own custom validation annotations.

Scenario: Ensure a string is a valid product SKU (e.g., starts with "SKU-" followed by 5 digits).

Steps:

Create the Annotation: Define your custom annotation (e.g., @ValidSKU).

Create the Validator: Implement ConstraintValidator<AnnotationType, FieldType>.

Associate: Link the annotation to the validator using @Constraint.

Code Example: Custom SKU Validation (Conceptual)

Java

// 1. Custom Annotation
import jakarta.validation.Constraint;
import jakarta.validation.Payload;
import java.lang.annotation.*;

@Documented
@Constraint(validatedBy = ValidSKUValidator.class) // Link to the validator
@Target({ ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER, ElementType.ANNOTATION_TYPE })
@Retention(RetentionPolicy.RUNTIME)
public @interface ValidSKU {
    String message() default "Invalid SKU format. Must start with 'SKU-' followed by 5 digits.";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// 2. Custom Validator Logic
import jakarta.validation.ConstraintValidator;
import jakarta.validation.ConstraintValidatorContext;

public class ValidSKUValidator implements ConstraintValidator<ValidSKU, String> {

    @Override
    public boolean isValid(String sku, ConstraintValidatorContext context) {
        if (sku == null || sku.trim().isEmpty()) {
            return false; // Or true, depending on whether null/empty is allowed by other annotations like @NotBlank
        }
        return sku.matches("^SKU-\\d{5}$");
    }
}

// 3. Using the Custom Annotation in DTO
public class ProductRequestDTO {
    // ... other fields ...

    @ValidSKU
    @NotBlank(message = "SKU is required")
    private String sku;

    // ... getters/setters for sku ...
}
