What multithreading means
Multithreading means a program does more than one task at the same time. In Java, this helps your app stay fast and responsive when work can be split into smaller parts.

Easy real-world examples
Restaurant kitchen: One cook makes rice, another makes curry, another packs the order.

ATM booth: One part checks balance, another updates transaction history, another sends SMS.

E-commerce app: One thread loads product details, another fetches reviews, another calculates delivery time.

YouTube app: One thread plays video, another buffers next part, another loads comments.

Bank system: One thread handles deposit, another handles withdrawal, another updates account summary.

Why it matters
Without multithreading, the app may wait for one task to finish before starting the next. With multithreading, tasks can overlap, so the app feels faster and handles more users.

One-line interview answer
Multithreading is a way to run multiple tasks concurrently in the same program to improve performance and responsiveness.

Very simple Java example idea
Imagine a food app:

Thread 1: load menu.

Thread 2: fetch restaurant ratings.

Thread 3: calculate delivery estimate.

Race condition
A race condition happens when two or more threads access the same shared data at the same time, and the final result depends on who finishes first. This can cause wrong values or inconsistent behavior in real applications.

Real-world example
Imagine two people trying to book the last train ticket at the same time.

Person A checks that 1 ticket is available.

Person B checks the same thing at almost the same time.

Both try to book it.

But only one ticket exists.

That is a race condition: the result depends on timing.




Thread safety
Thread safety means a class, method, or code block works correctly even when many threads use it at the same time. In practice, it prevents data corruption and inconsistent results when shared data is modified concurrently.



Real-world example
In a bank account:

Thread 1 deposits money.

Thread 2 withdraws money.

If both update the balance without protection, the final balance may become wrong.



Synchronization
Synchronization is a way to make sure only one thread at a time can access a critical section of code. It is one of the main ways Java prevents thread interference and memory consistency problems.

Real-world example
Think of a single-person washroom:

Only one person can enter at a time.

Others must wait.

That is how synchronized works for shared code.

Simple Java idea

class Counter {

        count++;
    }

    public int getCount() {
        return count;
    }
}

18) What is Semaphore?
A semaphore controls how many threads can access a resource at the same time.

It is useful for limiting concurrency, like restricting database connections.



A good real-time example is a restaurant with only 3 billing counters. At most 3 customers can pay at the same time; everyone else waits until one counter becomes free.

How it maps

A service becomes slow only under high load. What do you check first?
Check thread contention, blocked threads, queue buildup, and slow locks using thread dumps and profiling. High load often exposes synchronization bottlenecks.




DEADLOCK:
Thread 1 locks accountA and then tries to lock accountB.

Thread 2 locks accountB and then tries to lock accountA.




 How do you actually avoid a Deadlock?
To prevent a deadlock, you don't just "add a semaphore." Instead, you must follow strict design rules:

Acquire Locks in a Strict Order: All threads must request resources in the exact same sequence. For example, both Thread 1 and Thread 2 must grab Milk first, and only then try to grab Cookies.

Use Timeouts: Instead of waiting forever (acquire()), threads should use a timeout (tryAcquire(5, TimeUnit.SECONDS)). If they can't get the resource in 5 seconds, they give up and release what they are holding so other threads can move forward.








Your background email job is slowing down user requests. What do you do?
Move the job to a separate executor or async pipeline so user-facing threads are not blocked. Use timeouts and bounded queues to protect the main service.









 How do you design a thread-safe service?
Use immutable objects where possible.

Protect shared state with locks or concurrent structures.

Keep critical sections small.

Avoid shared mutable state unless necessary.

 Primary index
A primary index is built on the primary key, and the table data is usually stored in that order. It is fast for finding a row by its unique ID.

Example:

users(id) where id is the primary key.

Searching a customer by customer_id in an order system.

2. Secondary index
A secondary index is created on a non-primary column. It helps search by fields that are frequently queried but not the main key.

Example:

users(email)

Searching orders by status = 'PENDING' in an e-commerce app.

3. Clustered index
A clustered index determines the physical order of rows in the table. Usually, there can be only one clustered index per table.

Example:

Orders table clustered on order_date.

Good for recent-order lookups and range queries like “last 7 days orders.”

4. Non-clustered index
A non-clustered index stores pointers to the data rows instead of arranging the table itself. A table can have multiple non-clustered indexes.

Example:

Index on phone_number.

Index on city for customer search filters.

5. Unique index
A unique index prevents duplicate values in a column or set of columns. It is used when a field must remain distinct.

Example:

email in a login system.

username in an authentication service.

6. Composite index
A composite index is made from more than one column. It works best when queries filter on those columns together and in the same order.

Example:

(country, state, city)

(customer_id, order_date) for order history lookup.

7. Full-text index
A full-text index is used for searching text content efficiently. It is common in search features where users type words or phrases.

Example:

Blog search by title or content.

Product description search in an e-commerce site.

8. Hash index
A hash index is best for exact-match lookups. It is not good for range queries or sorting.

Example:

Looking up api_key = 'abc123'.

Session token validation.

9. Bitmap index
A bitmap index is useful when a column has low cardinality, meaning few distinct values. It works well in analytics systems.

Example:

gender, status, or payment_type.

Reports like “count all active users in a region.”

Livelock
Threads keep changing state, but no one makes progress.

Example:

Two threads keep retrying politely to avoid conflict.

Both keep stepping aside, but neither passes.

Real life:

Two people in a corridor both move left, then both move right, and keep repeating.


Starvation
One thread keeps waiting because other threads always get priority.

Example:

A low-priority background job never gets CPU time.

High-priority jobs keep coming, so it never runs.

Real life:

A regular customer at a restaurant keeps getting skipped because VIP customers keep arriving.



Atomic classes
Atomic classes help you do updates safely without using full synchronization. They are very useful for simple counters and flags.

Real-world example
Think of a digital turnstile counter at a stadium:

many people enter,

no two people should overwrite the same count.

Interview line
Use atomic classes when multiple threads update a simple shared value like a counter, and you want thread safety with better performance.

What is Kafka?
Kafka is a messaging system used to send data between applications.

Why use Kafka?
It helps services talk to each other quickly and safely.

What is a topic?
A topic is a place where messages are stored.

What is a producer?
A producer sends messages to Kafka.

What is a consumer?
A consumer reads messages from Kafka.

What is a partition?
A partition splits a topic into smaller parts for speed and scaling.

What is consumer group?
A consumer group is a set of consumers sharing the work.

What is offset?
Offset is the position of a message in a partition.

Real-world scenario questions
If messages are processed twice, what do you do?
Use idempotency or duplicate check.

If consumer lag is high, what do you check?
Check slow processing, fewer partitions, or database slowness.

If messages are out of order, what do you do?
Use the same key like orderId so related messages go to the same partition.

If one consumer is overloaded, what do you do?
Add more partitions or more consumers.

If a message keeps failing, what do you do?
Send it to a DLQ and continue other messages.

If producer becomes slow, what do you check?
Check network, broker load, batching, and acknowledgements.

“Partitioning helps Kafka scale.”

“Consumer groups help share load.”

“DLQ is used for bad messages.”

Answer: Kafka uses replication, so another broker takes over the partition leader.

Answer: Use idempotent producers, transactions, and dedup logic in consumers.
