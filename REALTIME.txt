NESTED propagation = savepoint under the hood = partial rollback without killing the whole transaction.



@Transactional
public void doAllInserts() {

    orderRepository.save(new Order("Insert-1", 1));
    orderRepository.save(new Order("Insert-2", 2));
    orderRepository.save(new Order("Insert-3", 3));

    // SAVEPOINT created before entering this method
    try {
        riskyInsertService.doRiskyInserts(); // 4, 5, 6 all inside
    } catch (RuntimeException e) {
        // 4, 5, 6 all rolled back to savepoint
        // outer tx still alive
        System.out.println("Rolled back 4,5,6: " + e.getMessage());
    }

    // Continues from here
    orderRepository.save(new Order("Insert-7",  7));
    orderRepository.save(new Order("Insert-8",  8));
    orderRepository.save(new Order("Insert-9",  9));
    orderRepository.save(new Order("Insert-10", 10));

    // Final commit — 1,2,3,7,8,9,10 saved
}

Core Concept: One file = One lock
Use a ConcurrentHashMap<String, ReentrantLock> where the key is the filename/path and the value is a lock for that file.

How it works step by step:

When Thread A wants to access report.pdf, it looks up the lock for report.pdf in the map
It acquires that lock — now Thread A owns it
Thread B comes in for the same report.pdf — it tries to acquire the same lock and blocks/waits
Thread C comes in for invoice.pdf — it gets a different lock, so it proceeds immediately ✅
When Thread A finishes, it releases the lock — Thread B unblocks


In Spring Boot context:

Create a singleton Spring @Service that holds this ConcurrentHashMap
Since it's a singleton, all threads across all requests share the same map
Use computeIfAbsent to safely create a lock if one doesn't exist yet for that file
Always release the lock in a finally block to avoid deadlocks

Spring Cloud Resilience4j - All Concepts
1. Circuit Breaker 🔌
Most important pattern - stops calling failing service, returns fallback

Aspect	Details
What it does	When service fails → circuit opens → stop sending requests 
States	CLOSED (normal) → OPEN (failing) → HALF_OPEN (testing) 
Why use	Prevent cascading failures in microservices 
Real-Time Usage (Your Project):

java
// Configuration
@Configuration
public class ResilienceConfig {

    @Bean
    public CircuitBreakerFactory circuitBreakerFactory() {
        return new Resilience4JCircuitBreakerFactory();
    }
}

// Service with Circuit Breaker
@Service
public class DriverNotificationService {

    @Autowired
    private PushNotificationService pushService;

    @CircuitBreaker(name = "push-notification", fallbackMethod = "fallbackNotification")
    public void sendToDriver(String driverId, String message) {
        // Call push notification service (may fail)
        pushService.sendPush(driverId, message);
    }

    // Fallback method when circuit opens
    public void fallbackNotification(String driverId, String message, Exception ex) {
        // Use SMS instead when push fails
        smsService.sendToDriver(driverId, message);
    }
}

2. Retry 🔄
Automatically retry failed request - useful for transient errors

Aspect	Details
What it does	Failed request → retry N times with delay 
Why use	Transient errors (network timeout) → retry succeeds 
Config	Max retries, retry interval, exception filter 
Real-Time Usage (Your Project):

java
// Configuration
@Bean
public RetryRegistry retryRegistry() {
    RetryConfig config = RetryConfig.custom()
        .maxAttempts(3)
        .waitDuration(Duration.ofSeconds(2))
        .retryOn(exception -> exception instanceof HttpRequestTimeoutException)
        .build();
    return RetryRegistry.of(config);
}

This flowchart illustrates a scalable and error-resilient architecture for validating 100 items, each requiring 1,500 backend validations over 5 minutes. The core principle is asynchronous processing using a message queue.
