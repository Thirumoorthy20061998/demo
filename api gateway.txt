Spring Boot API Gateway Real-Time Interview Q&A
1. Core Concepts & Architecture
Q: What is the primary role of an API Gateway in a microservices environment, and which Spring framework component typically fills this role?

A: The API Gateway acts as a single entry point for all client requests, abstracting the internal microservice architecture. Its primary purposes are request routing to the correct service instances and handling cross-cutting concerns like authentication, rate limiting, and monitoring. The standard Spring component is Spring Cloud Gateway (SCG).


Q: Why is Spring Cloud Gateway chosen over older solutions like Netflix Zuul for new projects? What's its architectural advantage?

A: SCG is built on Spring Boot and Spring WebFlux, making it fully non-blocking and reactive by leveraging the Netty server (instead of the Servlet API/Tomcat). This reactive nature allows it to handle a much higher volume of concurrent connections with fewer threads, resulting in superior performance and lower resource consumption compared to Zuul's older, blocking model.


Q: Explain the three fundamental components that define a Route in Spring Cloud Gateway.

A: A Route is defined by:

URI (Destination): The target location of the request, often pointing to a logical service name like lb://USER-SERVICE for service discovery.

Predicates: A set of conditions (Predicate<ServerWebExchange>) that must be true for the route to be matched. Examples include matching the request path, method (GET/POST), or header values.

Filters: Modifications applied to the request or response. These can run before sending the request downstream (pre-filter) or after receiving the response (post-filter).


2. Configuration and Service Interaction
Q: How do you configure a simple route to forward traffic for a service named USER-SERVICE?

A: Configuration is usually done in application.yml. You must use the lb:// prefix for load balancing and service discovery:


spring:
  cloud:
    gateway:
      routes:
      # --- ORDER SERVICE ROUTE ---
      - id: order_route
        uri: lb://ORDER-SERVICE
        predicates:
        - Path=/api/v1/orders/** # Matches any path starting with /api/v1/orders/
      # --- PAYMENT SERVICE ROUTE ---
      - id: payment_route
        uri: lb://PAYMENT-SERVICE
        predicates:
        - Path=/api/v1/payments/** # Matches any path starting with /api/v1/payments/



Q: How does Spring Cloud Gateway integrate with Service Discovery?

A: It uses the DiscoveryClient abstraction (like Eureka or Consul). When a route's URI is prefixed with lb://SERVICE-NAME, SCG resolves the logical SERVICE-NAME to the physical IP and port of an available instance using the service registry. It then uses its built-in client-side load balancer to select one of the instances.




3. Real-Time Scenarios and Resilience
Q: We need to enforce a rate limit of 10 requests per second per user. How do you implement this in the Gateway?

A: You use the RateLimiter Gateway Filter Factory (specifically RequestRateLimiter):

Configure a rate limiter bean (often backed by Redis for distributed limits).

Define the Key Resolver bean, which specifies what to rate limit (e.g., extracting the user ID from a JWT token/Principal, or using the client's IP address).

Apply the filter to the route, setting the Replenish Rate and Burst Capacity

Q: Describe how the Gateway should handle security (Authentication and Authorization).A: The Gateway should act as a security filter layer.11 It typically uses a custom pre-filter to:Extract the security token (e.g., JWT) from the request.Validate the token (signature, expiration, and issuer).Authorize the request (check if the token has the necessary roles/scopes for the requested URI).If successful, forward the validated user principal/ID by adding it as a request header (e.g., X-User-ID) to the downstream service. If validation fails, it immediately rejects the request with a $401$ (Unauthorized) or $403$ (Forbidden).

Q: A downstream service is failing and timing out, causing cascading failures. How does Spring Cloud Gateway protect itself and other services from this?

A: By implementing the Circuit Breaker pattern using the Spring Cloud Circuit Breaker library (usually backed by Resilience4j).

The CircuitBreaker Filter is applied to the failing route.

It monitors calls to the downstream service. If the failure rate or latency exceeds a configured threshold, the circuit opens.

While the circuit is open, all new requests immediately return a fallback response without even trying to call the failing service, protecting the service and preventing a cascading failure.

4. Production Issues
Q: In a production environment, users complain about increased latency only when hitting the Gateway. Where would you start troubleshooting?

A: Since SCG is reactive, high latency often points to an issue that is causing blocking.

Filter Chain Analysis: Check custom-written Gateway Filters for any code performing synchronous, blocking I/O (e.g., JDBC calls, blocking REST calls). All I/O in a filter must be non-blocking and reactive.

Custom Logic: Examine custom logic for inefficient code, particularly involving complex operations on reactive streams.

Dependency Latency: Check if the security validation (e.g., talking to an external OAuth server) is slow.








The API Gateway and Service Discovery work together to provide dynamic routing and load balancing in a microservices architecture. Instead of the Gateway needing to be configured with the hardcoded IP addresses and ports of every microservice, it uses the service registry to find the services in real-time.

This setup is known as Server-Side Discovery, where the gateway acts as the client that queries the registry to resolve the destination service name.

How API Gateway Works with Service Discovery
The process involves three main components: Microservices, the Service Registry (e.g., Eureka, Consul), and the API Gateway (e.g., Spring Cloud Gateway).

Service Registration: When a microservice (e.g., ORDER-SERVICE) starts up, it registers its network location (IP address and port) and its logical service name (ORDER-SERVICE) with the Service Registry. It also sends heartbeats to keep the registration current.

Route Configuration: The API Gateway is configured with logical service names instead of physical addresses. For example, a route might specify the URI as lb://ORDER-SERVICE.

Dynamic Routing: When the Gateway receives an incoming client request, it performs the following steps:

It matches the request against its routing rules (predicates).

It sees the destination is a logical service name (lb://ORDER-SERVICE).

It queries the Service Registry for the available instances of ORDER-SERVICE.

The Registry returns a list of active IPs and ports for that service (e.g., 10.0.0.5:8081, 10.0.0.6:8081).

The Gateway's built-in client-side load balancer selects one of these instances (e.g., using Round Robin).

The request is forwarded to the selected instance.
