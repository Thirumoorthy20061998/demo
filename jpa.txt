
Setting Up and Initializing
git init

Usage: git init

Description: Initializes a new Git repository in the current directory.

git clone

Usage: git clone <repository-url>

Description: Downloads an existing repository from a remote source (like GitHub) to your local machine.

git config

Usage: git config --global user.name "Your Name" and git config --global user.email "your.email@example.com"

Description: Sets your user name and email for all future commits (the --global flag makes it apply to all your repos).

📝 Tracking Changes (The Workflow)
git status

Usage: git status

Description: Shows the status of changes as untracked, modified, or staged.

git add

Usage: git add <file-name> or git add .

Description: Stages changes from the working directory. The dot (.) stages all modified files.

git commit

Usage: git commit -m "Your commit message"

Description: Records the staged changes into the local repository with a descriptive message.

git diff

Usage: git diff or git diff --staged

Description: Shows the difference between the working directory and the staging area, or between the staging area and the last commit.

git log

Usage: git log

Description: Displays the history of commits for the current branch.

🌳 Branching and Merging
git branch

Usage: git branch (lists branches) or git branch <new-branch-name> (creates a branch)

Description: Lists all local branches, or creates a new one.

git checkout

Usage: git checkout <branch-name> or git checkout -b <new-branch-name>

Description: Switches to another branch. The -b option both creates and switches to a new branch.

git switch

Usage: git switch <branch-name>

Description: A newer, clearer alternative to git checkout for switching branches.

git merge

Usage: git merge <branch-name>

Description: Integrates changes from the specified branch into the current branch.

git branch -d

Usage: git branch -d <branch-name>

Description: Deletes the specified local branch (after it has been merged).

🔗 Collaborating (Remote Repositories)
git remote

Usage: git remote add origin <repository-url>

Description: Connects your local repository to a remote server, usually named origin.

git push

Usage: git push -u origin <branch-name>

Description: Uploads your local branch commits to the remote repository.

git pull

Usage: git pull origin <branch-name>

Description: Fetches and merges changes from the remote repository to your current local branch.

git fetch

Usage: git fetch origin

Description: Downloads commits, files, and refs from a remote repository into the local repository without merging them.

⏪ Undoing and Rewriting History
git reset

Usage: git reset <file-name> (unstage) or git reset --hard <commit-hash> (undo commits)

Description: Unstages a file, or completely undoes commits and discards all changes since that commit.

git revert

Usage: git revert <commit-hash>

Description: Creates a new commit that undoes the changes introduced by the target commit (safer for shared history).

git restore

Usage: git restore <file-name>

Description: Discards changes in the working directory for a specific file, returning it to its last committed state.

orphanRemoval=true:

Definition: This property is specifically used with @OneToMany and @OneToOne relationships. When set to true, if a child entity is removed from its parent's collection (i.e., it becomes "orphaned" because no parent references it), JPA will automatically delete that child entity from the database. It implies CascadeType.REMOVE for that specific scenario.


CascadeType.ALL:

Definition: This cascade type means that all persistence operations (like PERSIST, MERGE, REMOVE, REFRESH, DETACH) performed on the parent entity will automatically be applied to its associated child entities.

What are the four entity lifecycle states?	1. Transient: A new object, never persisted, not associated with a context, has no ID. 2. Managed (Persistent): The entity is in the context; changes are tracked and saved to the DB upon transaction commit (dirty checking). 3. Detached: The entity was once managed but its context/session is closed. Changes are not tracked. 4. Removed: The entity is managed but scheduled for deletion from the database.

What is SLF4J and why is it preferred over using Log4j or Logback directly in a Spring Boot application?	SLF4J is a logging facade (an abstraction layer), not an actual logging implementation. It allows you to write logging code that is implementation-agnostic. Spring Boot prefers it because you can swap out the actual logging framework (like Logback, Log4j2, or JUL) at any time by changing a dependency, without needing to modify a single line of application code.
