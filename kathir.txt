HashMap works using hashing to store key-value pairs. Internally it's an array of buckets, where each bucket holds key-value nodes.

How it works:

When you call put(key, value), it calculates key.hashCode() → finds a bucket index

Stores the key-value pair in that bucket

If two keys get the same index (collision), it uses a linked list (Java 7) or linked list + Red-Black Tree (Java 8+) to store multiple entries

get(key) calculates the hash, finds the bucket, traverses the list/tree, and returns the value using equals()

Static Methods (Java 8)
What they are: Methods with a body and the static keyword, belonging entirely to the interface itself.

Why they exist: To provide utility or helper methods directly related to the interface. (e.g., Collections.sort() could theoretically just be a static method on a Collection interface today).

Key Fact: They cannot be overridden by implementing classes. You also cannot call them using an object instance; you must call them using the interface name itself (e.g., MyInterface.myStaticMethod()).


Private Methods (Java 9)
What they are: Helper methods inside the interface marked as private.

Why they exist: Internal Code Reusability. If you have two different default methods in your interface that share 10 lines of the exact same logic, you can extract those 10 lines into a private method.

Key Fact: They are completely hidden. Implementing classes cannot see, call, or override them. They exist solely to be used by the default or static methods inside that specific interface.


Default Method (The Standard Perk): "Take a lunch break."
Ten years later, the company decides everyone gets a break. Instead of tearing up everyone's old contracts (breaking the code), they add a standard 30-minute break as a default. Everyone automatically gets it. But, if you are an executive, you can override it to take a 2-hour lunch instead.

Private Method (The Behind-the-Scenes Math): "Calculate tax deduction."
Let's say the company adds a default "Give Holiday Bonus" method. To figure out the bonus, they need to do some tax math. They put this math into a private section of the contract. The employee never sees it, interacts with it, or changes it—it just helps the contract do its own internal paperwork.

Static Method (The General Info): "Get Company Headquarters Address."
You don't need to track down a specific employee to find out where the company is located. That information belongs to the company as a whole. You just look at the master contract itself to find it.

What are Records in Java?
Introduced in Java 16, a Record is a special type of class designed solely to hold immutable data. By using a single line of code, Java automatically generates the constructor, getters, equals(), hashCode(), and toString() methods for you.
public record Employee(String name, int age, double salary) {}


Records vs. POJOs vs. DTOs
Think of them as having different jobs:

POJO (Plain Old Java Object): The most general term. Any Java class that doesn’t depend on a specific framework. It usually has private fields, getters, and setters, and its data can be changed (mutable).

DTO (Data Transfer Object): A design pattern used to ship data between systems or layers (like from a database to a frontend). DTOs are often immutable, but traditionally written as standard POJO classes.

Record: A language feature (not just a pattern). It is a structurally immutable carrier of data.

No. Records are strictly immutable. All fields are implicitly final. There are no setter methods, and you cannot change the data once the record is created.

@Service

    private ProductRepository productRepository;
    
    public Page<Product> getProducts(int page, int size, String sortBy) {
        // Page number is 0-based in Spring Data JPA [web:100]
        Pageable pageable = PageRequest.of(page - 1, size, Sort.by(sortBy));  // [web:100][web:104]

        Pageable pageable = PageRequest.of(page - 1, size);
        return productRepository.findByCategory(category, pageable);
    }
}

What is @Bean and how does it differ from @Component?
@Bean is used inside a @Configuration class to manually declare a bean. @Component is class-level and needs to be on the class itself (usually your own class). @Bean is used when:
• You don't own the class (e.g., third-party lib).
• You need runtime logic to configure the bean.



When Bean A is being constructed:
• Spring puts a factory into singletonFactories for A.
• If Bean B is requested during A's creation and B needs A, Spring calls the factory, moves the early ref to earlySingletonObjects, and gives it to B.
• A finishes construction and moves to singletonObjects.


Why constructor injection breaks: The factory can only be registered AFTER the constructor runs. If A's constructor requires B, and B's constructor requires A, neither can finish its constructor — there is no early ref to hand out. Spring throws BeanCurrentlyInCreationException immediately.
