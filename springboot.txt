1. What are the different bean scopes in Spring Boot?

Answer:
Spring provides 5 main bean scopes:

1. Singleton – Only one instance per Spring container (default scope).


2. Prototype – New instance every time the bean is requested.


3. Request – One instance per HTTP request (Web context).


4. Session – One instance per HTTP session (Web context).


5. Application – One instance per ServletContext (Web context).

Default bean scope in Spring.

Spring container creates only one instance of the bean and shares it across the application.
✅ Real-time example:
A service like UserService or OrderService used throughout the application.

   // Shared across application
}

Answer:

A new bean instance is created every time it’s requested from the container.
✅ Real-time example:
A bean that maintains user-specific data or temporary objects like EmailMessage or ReportGenerator.


@Component
@Scope("prototype")

   // Creates new instance for each request
}

Answer:
Use Prototype when:

The bean holds mutable state (changes per request).

Each thread or request needs a fresh instance.
✅ Example:
If you generate PDF reports or process images, you should not share the same object.

Creates a new bean for each HTTP request.

How to Create Beans
There are two primary ways to create beans in Spring Boot:

1. Using @Component and its Stereotypes (Automatic Detection)
This is the most common and recommended way. You mark a class with one of Spring's stereotype annotations, and Spring will automatically detect it and register it as a bean. The most generic annotation is @Component.


@Component: A generic stereotype for any Spring-managed component.

@Service: A specialized @Component for service-layer classes, which typically contain business logic.

@Repository: A specialized @Component for data-access layer classes (like DAOs), which handle database operations.

@Controller and @RestController: Specialized @Component for web-layer classes that handle HTTP requests.

Example:

Java

import org.springframework.stereotype.Service;

@Service
public class MyBusinessService {
    // This class is now a Spring bean
    public String performTask() {
        return "Task performed!";
    }
}
In this example, Spring will automatically create a single instance of MyBusinessService and manage it for you.

2. Using @Bean Annotation (Explicit Configuration)
You can also explicitly define a bean using the @Bean annotation within a class marked with @Configuration. This approach is useful when you need to create a bean from a third-party library or when the bean creation logic is more complex.

@Configuration: This annotation marks a class as a source of bean definitions.

@Bean: This method-level annotation tells Spring that the method returns an object that should be registered as a bean.

Example:

Java

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class AppConfig {

    @Bean
    public MyBusinessService myBusinessService() {
        // This method returns the object to be registered as a bean
        return new MyBusinessService();
    }
}

What's the difference between an embedded server and an external application server in the context of Spring Boot?

Answer: An embedded server (like Tomcat, Jetty, or Undertow) is included directly within the Spring Boot application's executable JAR file. Design & Architecture Questions
When would you choose synchronous communication (REST/HTTP) over asynchronous (Messaging/Kafka) between microservices, and vice versa?

Answer:

Synchronous (REST/HTTP):

When to use: Immediate response required, tight coupling is acceptable (e.g., request-response for UI updates), simple point-to-point communication, low volume of requests, idempotent operations.

Pros: Simpler to implement initially, easy to debug with standard HTTP tools.

Cons: Tightly coupled, blocking, single point of failure (without resilience patterns), harder to scale.

Asynchronous (Messaging/Kafka):

When to use: Decoupled services, event-driven architectures, high throughput/volume, long-running processes, fault tolerance (messages can be retried), audit trails.

Pros: Highly decoupled, resilient (message brokers can store messages), scalable, enables eventual consistency.

Cons: More complex to implement and debug, requires managing message brokers, eventual consistency can be a challenge for users.

What is the "Database per Service" pattern in microservices, and what are its implications for data consistency? How does Spring Boot fit in?

Answer: The "Database per Service" pattern means each microservice owns its private database. It cannot access another service's database directly.

Implications:

Autonomy: Services are highly independent, allowing for technology choices (polyglot persistence) and independent scaling/deployment.

Data Consistency: Achieving consistency across services for transactions is challenging. Traditional 2PC is avoided. This leads to the Saga pattern and eventual consistency, where data across services might be temporarily inconsistent but eventually converges.

Spring Boot's Fit: Spring Boot facilitates this by making it easy to configure separate data sources for each microservice using spring-boot-starter-data-jpa (or other Spring Data modules for NoSQL). Each microservice can then manage its own schema and data access layer independently.

You encounter a java.lang.OutOfMemoryError in your Spring Boot application. What steps would you take to diagnose and resolve it?

Answer:

Identify the type: Is it Heap space (memory leak, too small heap) or PermGen/Metaspace (class loading issues)?

Increase Heap (Temporary Fix): Increase -Xmx JVM argument. (Not a permanent solution but might give breathing room).

Analyze Heap Dump: Use jmap or -XX:+HeapDumpOnOutOfMemoryError to generate a heap dump. Analyze with tools like Eclipse Memory Analyzer Tool (MAT) or VisualVM to identify memory leaks, large objects, and dominant retainers.

Monitor with Actuator/JMX: Use Spring Boot Actuator's /heapdump endpoint or JMX tools to monitor memory usage in real-time.

Code Review: Look for inefficient data structures, unclosed resources (streams, connections), caching issues, or large object creation.

Connection Pool Sizing: If it's related to database connections, check connection pool configuration.

Profile Application: Use profilers (e.g., JProfiler, YourKit, VisualVM) to pinpoint memory hotspots and object allocations.

Spring Security Enhancements
What is the role of SecurityFilterChain in Spring Security 6 (and later), and how does it differ from the deprecated WebSecurityConfigurerAdapter?

Answer: In Spring Security 6+, WebSecurityConfigurerAdapter has been deprecated. The new approach is to define security configurations using SecurityFilterChain beans.

WebSecurityConfigurerAdapter (Deprecated): This was an abstract class you would extend and override methods (like configure(HttpSecurity)) to define your security rules. It often led to complex inheritance hierarchies and less flexible configurations.

SecurityFilterChain (New): You now declare SecurityFilterChain objects as @Beans in your @Configuration class. Each SecurityFilterChain defines a specific set of security rules (e.g., for certain request matchers, or specific paths) that Spring Security applies. This provides:

Flexibility: You can have multiple SecurityFilterChain beans, each applying different security rules to different parts of your application based on request matchers.

Clarity: It's often easier to reason about specific security concerns isolated in individual bean definitions.

Functional Style: Aligns better with modern Spring's functional programming leanings.

Example:

Java

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(authorize -> authorize
                .requestMatchers("/public/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .formLogin(Customizer.withDefaults()); // Or .httpBasic(Customizer.withDefaults());
        return http.build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        UserDetails user = User.withDefaultPasswordEncoder()
            .username("user")
            .password("password")
            .roles("USER")
            .build();
        return new InMemoryUserDetailsManager(user);
    }
}
Discuss the common authentication and authorization mechanisms you can implement using Spring Security in a Spring Boot REST API.

Answer:

Authentication (Verifying Identity):

Basic Authentication: Simplest, credentials sent in HTTP header. Suitable for internal APIs or when combined with HTTPS.

Token-based Authentication (JWT/OAuth2): Most common for REST APIs and microservices.

@RestClientTest:

Purpose: For testing REST clients (like RestTemplate or WebClient).

When to use: To test how your application consumes external REST APIs, mocking the responses from the external service.

@WebFluxTest:

Purpose: Similar to @WebMvcTest but for Spring WebFlux controllers.

When to use: For testing your reactive web controllers.

How would you implement asynchronous processing in a Spring Boot application without using Spring WebFlux?

Answer: You can use Spring's @Async annotation.

Enable Asynchronous Processing: Add @EnableAsync to one of your @Configuration classes.

Annotate Methods: Annotate methods that should run asynchronously with @Async.

Return Types: Asynchronous methods can return void or java.util.concurrent.Future (or CompletableFuture for more powerful composition).

Describe how to properly shut down a Spring Boot application gracefully.

Answer: Graceful shutdown allows an application to finish processing current requests and release resources before terminating, preventing data loss or client errors.

Default Behavior: Spring Boot applications listen for shutdown signals (like SIGTERM from Docker/Kubernetes or Ctrl+C).

Configuration:

server.shutdown=graceful (Spring Boot 2.3+): This property configures the embedded web server (Tomcat, Jetty, Undertow) to pause accepting new connections and allow existing requests to complete within a configurable timeout.

spring.lifecycle.timeout-per-shutdown-phase: Configures the maximum time in milliseconds that Spring Boot will wait for each shutdown phase (e.g., web server shutdown, bean destruction).

Implications for Microservices: In a containerized environment (Docker, Kubernetes):

The orchestration platform (Kubernetes) sends a SIGTERM signal.

The application starts its graceful shutdown.

Kubernetes waits for a terminationGracePeriodSeconds (default 30s). If the app doesn't shut down gracefully within this period, Kubernetes sends a SIGKILL.

Best Practices:

Ensure your application code handles interruptions and releases resources (database connections, message consumers).

Use server.shutdown=graceful.

Monitor logs during shutdown to confirm proper termination.

Configure appropriate terminationGracePeriodSeconds in Kubernetes deployments.

S - Single-responsibility Principle (SRP)
In Spring Boot, you follow this by creating focused classes with a single purpose. For example, a UserController should only handle HTTP requests and responses, a UserService should contain the business logic, and a UserRepository should only deal with database interactions. Spring's component model and annotations (@Controller, @Service, @Repository) naturally guide you toward this separation of concerns.

O - Open-closed Principle (OCP)
You can achieve this in Spring Boot by using interfaces and dependency injection. Instead of writing conditional logic to handle different cases, you define an interface and create multiple implementations. For example, you might have a PaymentService interface with CreditCardPaymentService and PayPalPaymentService implementations. Spring's dependency injection will handle which implementation to use at runtime, so you can add a new payment method without changing the existing PaymentService code.

L - Liskov Substitution Principle (LSP)
This principle is upheld by correctly using interfaces and inheritance. If you have a superclass or an interface, any class that extends it or implements it should be able to be used interchangeably without causing unexpected behavior. Spring's framework helps enforce this by working with a polymorphic model, where you can inject and use any subclass as long as it adheres to the superclass's contract.

I - Interface Segregation Principle (ISP)
Spring Boot helps you adhere to this by encouraging the use of small, specific interfaces. For instance, instead of a single large CustomerService interface with dozens of methods, you might create separate interfaces like CustomerCreationService and CustomerAnalyticsService. Your classes then only implement the interfaces they need, avoiding unnecessary dependencies.

D - Dependency Inversion Principle (DIP)
This is arguably where Spring Boot shines the most. Its powerful dependency injection (DI) container is the primary tool for this principle. You code against an interface (abstraction) and let the Spring container inject the concrete implementation at runtime. This inverts the typical dependency from a high-level module depending on a low-level module to both depending on an abstraction, leading to a loosely coupled and highly testable application.















Here's why and when you'd choose Spring WebFlux over the traditional Spring MVC:

Non-Blocking I/O for High Concurrency:

Spring MVC (Traditional): Uses a "one thread per request" model. When a request needs to do an I/O operation (like fetching data from a database), that thread blocks (waits) until the I/O is complete. If you have many concurrent requests, your server can quickly run out of threads, leading to slow responses or even crashes.

Spring WebFlux (Reactive): Uses a small, fixed number of threads (often matching CPU cores) that never block. When an I/O operation is initiated, the thread registers a "callback" (instructions for what to do when the I/O is done) and immediately becomes free to handle other requests. When the I/O finishes, the callback is executed. This allows a few threads to manage a vast number of concurrent connections efficiently.

Here's the short, full process of JWT creation, processing, and validation:

Creation (Login):

User sends username/password to server.

Server verifies credentials.

If valid, server creates a JWT by taking a Header (alg, type) and a Payload (user ID, roles, expiration time), combining them, and then digitally signing this combination with a secret key.

Server sends the JWT back to the client.

Processing & Validation (Subsequent Requests):

Client stores the JWT.

For every subsequent request to protected resources, the client attaches the JWT in the Authorization: Bearer <token> header.

Server receives the request.

A security filter/interceptor on the server extracts the JWT.

This filter then validates the JWT:

Verifies the signature using the secret key (ensuring it hasn't been tampered with and was issued by this server).

Checks expiration time.

(Optional) Checks other claims like issuer or audience.

If valid, the filter extracts user details (e.g., user ID, roles) from the JWT's payload and sets up the security context for the current request.
