Spring Boot Annotation-Based Interview Questions
Basic & Core Annotations
What is the purpose of @Controller and @RestController annotations? How do they differ?

Answer:

@Controller: Marks a class as a Spring MVC controller. It's typically used in traditional web applications that render views (e.g., using Thymeleaf, JSP) or return string views. If you want to return raw data (like JSON/XML), you would need to combine it with @ResponseBody on individual methods or the class level.

@RestController: A convenience annotation that combines @Controller and @ResponseBody. It's specifically designed for building RESTful web services. Methods within a @RestController automatically serialize their return values into the HTTP response body (e.g., JSON, XML), eliminating the need for @ResponseBody on every method.

Difference: RestController implies ResponseBody on all handler methods, making it ideal for pure REST APIs. Controller is more general-purpose and requires ResponseBody for data, or returns views by default.

Explain the three core annotations combined by @SpringBootApplication.

Answer: @SpringBootApplication is a meta-annotation that simplifies the setup of a Spring Boot application by combining three common annotations:

@Configuration: Tags the class as a source of bean definitions. It indicates that the class can declare @Bean methods, which contribute to the Spring application context.

@EnableAutoConfiguration: Enables Spring Boot's auto-configuration mechanism. This tells Spring Boot to automatically configure your application based on the JAR dependencies present in your classpath (e.g., if H2 is on classpath, it configures a data source).

@ComponentScan: Configures component scanning. By default, it scans the package of the @SpringBootApplication class and all its sub-packages for Spring components (like @Component, @Service, @Repository, @Controller, @RestController, @Configuration) and registers them as beans.

What is @Autowired used for, and what are its common types of injection? Which type is generally recommended?

Answer: @Autowired is a Spring annotation used for automatic dependency injection. Spring's IoC container automatically resolves and injects collaborating objects (dependencies) into the annotated field, constructor, or setter method.

Types of Injection:

Field Injection (Least Recommended): Directly annotating a field. java @Autowired private MyService myService; 

Pros: Concise.

Cons: Hides dependencies (not clear from constructor), makes unit testing harder without Spring context (requires reflection), can lead to circular dependency issues being harder to spot.

Setter Injection: Annotating a setter method. java private MyService myService; @Autowired public void setMyService(MyService myService) { this.myService = myService; } 

Pros: Allows for optional dependencies (if not marked @Required), mutable (can change dependency after creation).

Cons: Dependencies aren't immediately clear from the constructor, can still be modified.

Constructor Injection (Most Recommended): Annotating the constructor. (Since Spring 4.3, @Autowired is optional if there's only one constructor). java private final MyService myService; public MyController(MyService myService) { this.myService = myService; } 

Pros: Immutability (dependencies are final), Testability (easy to instantiate and test outside Spring), Clarity (dependencies are explicit), Circular Dependency Detection (Spring detects them at startup).

Cons: Can lead to "constructor explosion" if too many dependencies (often a sign of a class doing too much).

Explain @Service, @Repository, and @Component. How do they relate?

Answer: These are Spring stereotype annotations used to mark classes as Spring-managed components, making them eligible for component scanning and dependency injection.

@Component: A generic stereotype for any Spring-managed component. It's the base annotation for all other stereotypes.

@Service: A specialization of @Component used to indicate that a class holds business logic. It doesn't add new technical capabilities over @Component but improves readability and clarifies the role of the class in the architecture.

@Repository: A specialization of @Component used to indicate that a class performs data access operations. It also provides automatic translation of persistence-related exceptions (e.g., JDBC SQLException to Spring's DataAccessException hierarchy).

Relationship: @Service and @Repository are meta-annotated with @Component. This means that any class annotated with @Service or @Repository is also a Spring component and will be picked up by component scanning. They offer semantic clarity and sometimes provide specific features (like exception translation for @Repository).

What is @Configuration used for in Spring Boot?

Answer: The @Configuration annotation marks a class as a source of bean definitions for the Spring application context. It's a key part of Java-based configuration in Spring. Inside a @Configuration class, you typically define methods annotated with @Bean. Each such method produces a bean that will be managed by the Spring IoC container. This allows you to programmatically define beans and their dependencies, replacing the need for XML configuration.

How do you define a Spring bean using annotations? Give an example.

Answer: You define a Spring bean using the @Bean annotation within a class that is annotated with @Configuration.

Example:

Java

@Configuration
public class AppConfig {

    @Bean
    public MyService myService() {
        // Here you can set up dependencies or properties for MyService
        return new MyService();
    }

    @Bean
    public MyComponent myComponent(MyService service) { // MyService is injected here
        return new MyComponent(service);
    }
}
In this example, myService and myComponent are both registered as beans in the Spring application context.

Configuration & Properties
Explain @Value for injecting properties. When would you prefer @ConfigurationProperties over @Value?

Answer:

@Value: Used to inject single property values from application.properties, application.yml, environment variables, or system properties directly into fields or method parameters. It uses Spring Expression Language (SpEL).

Example: @Value("${app.name}") private String appName;

@ConfigurationProperties: Used to bind external properties (often with a common prefix) to fields of a type-safe configuration bean.

Example:

Java

@Configuration
@ConfigurationProperties(prefix = "app.security")
public class SecurityProperties {
    private String jwtSecret;
    private long tokenExpirationMs;
    // getters and setters
}
In application.yml:

YAML

app:
  security:
    jwt-secret: mySuperSecretKey
    token-expiration-ms: 3600000
Preference:

@Value: Prefer for injecting single, standalone properties.

@ConfigurationProperties: Prefer when you have a group of related properties, especially those with a hierarchical structure. It provides:

Type-safety: Properties are automatically converted to the correct type.

Better Organization: All related properties are in one class.

Validation: Can be combined with @Validated for JSR-303 Bean Validation.

IDE Support: Better auto-completion and documentation hints in IDEs.

How do @Profile annotations work in Spring Boot? Give an example of its usage.

Answer: @Profile is used to define beans or @Configuration classes that should only be active when a specific Spring profile (or set of profiles) is active. This allows you to segregate configuration for different environments (e.g., development, QA, production) without changing the code.

Usage:

On @Configuration classes: java @Configuration @Profile("dev") public class DevConfig { @Bean public MyDevService devService() { return new MyDevService("Dev environment"); } } 

On @Component or @Service beans: java @Service @Profile({"prod", "qa"}) public class RealPaymentGateway implements PaymentGateway { // ... } @Service @Profile("dev") public class MockPaymentGateway implements PaymentGateway { // ... } 

Activation: Profiles can be activated via:

application.properties: spring.profiles.active=dev

Command-line argument: --spring.profiles.active=dev

Environment variable: SPRING_PROFILES_ACTIVE=dev

Web & REST Annotations
What is @RequestMapping used for, and how has @GetMapping, @PostMapping, etc., simplified its usage?

Answer:

@RequestMapping: The original Spring MVC annotation used to map HTTP requests to handler methods in controllers. It can be used at both class and method levels and supports specifying HTTP method (GET, POST, etc.), path, consumes, produces, headers, and params.

Example: @RequestMapping(value = "/users", method = RequestMethod.GET)

Simplification with @GetMapping, @PostMapping, etc.: These are composed annotations (meta-annotations) introduced in Spring 4.3 that are specializations of @RequestMapping. They pre-define the method attribute, making code more concise and readable.

@GetMapping is @RequestMapping(method = RequestMethod.GET)

@PostMapping is @RequestMapping(method = RequestMethod.POST)

@PutMapping is @RequestMapping(method = RequestMethod.PUT)

@DeleteMapping is @RequestMapping(method = RequestMethod.DELETE)

@PatchMapping is @RequestMapping(method = RequestMethod.PATCH)

Advantage: Improved readability and directness, clearly indicating the HTTP method.

Explain the purpose of @RequestParam, @PathVariable, and @RequestBody.

Answer: These annotations are used to bind incoming HTTP request data to method parameters in Spring MVC/WebFlux controllers.

@RequestParam: Binds a request parameter (from the URL query string, e.g., /api/items?id=123) to a method parameter.

Example: public Item getItemById(@RequestParam("id") Long itemId)

Useful for optional parameters, filtering, pagination.

@PathVariable: Binds a URI template variable (part of the URL path, e.g., /api/items/123) to a method parameter.

Example: public Item getItemById(@PathVariable("id") Long itemId)

Useful for identifying specific resources.

@RequestBody: Binds the entire content of the HTTP request body to a method parameter. Spring automatically converts the body (typically JSON or XML) into a Java object using HTTP message converters (like Jackson for JSON).

Example: public ResponseEntity<Void> createItem(@RequestBody Item item)

Used for POST, PUT, PATCH requests where data is sent in the body.

How do @ResponseStatus and @ResponseBody contribute to REST API responses?

Answer:

@ResponseBody: This annotation indicates that the return value of a controller method should be bound directly to the HTTP response body. Spring uses HTTP message converters to serialize the Java object into a suitable format (e.g., JSON, XML) and sends it back to the client. It's implicitly included in @RestController.

@ResponseStatus: This annotation marks a method or an exception class with an HTTP status code. When a method annotated with @ResponseStatus completes successfully, the specified HTTP status code is sent. When an exception class is annotated, that HTTP status code is returned if an instance of that exception is thrown.

Example on a method: @ResponseStatus(HttpStatus.CREATED) (often combined with @PostMapping methods).

Example on a custom exception: java @ResponseStatus(HttpStatus.NOT_FOUND) public class ResourceNotFoundException extends RuntimeException { // ... } 

What is @ControllerAdvice and @RestControllerAdvice used for in global exception handling?

Answer: These annotations are specializations of @Component that allow you to centralize cross-cutting concerns, particularly global exception handling, across multiple controllers.

@ControllerAdvice: A stereotype annotation that allows a class to provide global @ExceptionHandler, @InitBinder, or @ModelAttribute methods that apply to all @Controller classes (or a subset specified). It's typically used in combination with @ResponseBody if you want to return JSON/XML for error responses.

@RestControllerAdvice: A convenience annotation that combines @ControllerAdvice and @ResponseBody. It's specifically designed for global exception handling in RESTful APIs, as it automatically serializes the return value of exception handler methods into the HTTP response body.

Purpose: They prevent boilerplate exception handling code in every controller and provide a consistent error response format across your application.

Data Access Annotations
What are the key annotations used in Spring Data JPA repositories?

Answer:

@Repository: (As discussed before) Marks the interface/class as a data access object, enabling exception translation.

@Entity: (JPA annotation) Marks a Java class as an entity that maps to a table in a relational database.

@Id: (JPA annotation) Marks the primary key field of an entity.

@GeneratedValue: (JPA annotation) Specifies that the primary key value is automatically generated by the database.

@Column: (JPA annotation) Specifies the mapped column for a persistent property or field.

@Table: (JPA annotation) Specifies the primary table for the annotated entity.

@Transactional: (Spring annotation) Ensures that a method (or class) runs within a transaction. Spring manages the transaction lifecycle (begin, commit, rollback). Used heavily in service layers or directly on repository methods for write operations.

@Query: Used in Spring Data JPA repository interfaces to define custom queries using JPQL (Java Persistence Query Language) or native SQL.

Example: @Query("SELECT u FROM User u WHERE u.email = ?1") User findByEmailAddress(String email);

@Query(value = "SELECT * FROM users WHERE email = ?1", nativeQuery = true)


What is @MockBean used for in Spring Boot tests?

Answer: @MockBean is a Spring Boot testing annotation (from spring-boot-test) used to add Mockito mocks to the Spring ApplicationContext.

Purpose:

Replace existing beans: If a bean of a certain type already exists in the context, @MockBean will replace it with a Mockito mock.

Add new mocks: If no bean of that type exists, @MockBean will add a new mock bean to the context.

Benefit: It allows you to isolate the component being tested by mocking its dependencies within the Spring context, ensuring that your tests are focused and repeatable without relying on real external services or databases. It's particularly useful with sliced tests like @WebMvcTest or @DataJpaTest.

Example: To test a controller, you'd mock its dependent service:

Java

@WebMvcTest(UserController.class)
public class UserControllerTest {
    @Autowired private MockMvc mockMvc;
    @MockBean private UserService userService; // Mocks UserService in the context

    @Test
    void testGetUserById() throws Exception {
        given(userService.findById(1L)).willReturn(new User(1L, "Test User"));
        mockMvc.perform(get("/users/1"))
               .andExpect(status().isOk())
               .andExpect(jsonPath("$.name").value("Test User"));
    }
}
Advanced Annotations & Concepts
Explain @ConditionalOnClass and @ConditionalOnMissingBean. How are they crucial for Spring Boot's auto-configuration?

Answer: These are part of Spring's @Conditional annotations, which are fundamental to Spring Boot's auto-configuration mechanism. They allow beans or configuration classes to be included or excluded based on certain conditions at runtime.

@ConditionalOnClass(value = MyClass.class): The annotated bean or @Configuration class will only be loaded if MyClass (or all specified classes) is present on the application's classpath.

Use Case: Spring Boot's DataSourceAutoConfiguration might use this to only configure a DataSource if JDBC driver classes are found.

@ConditionalOnMissingBean(value = MyBean.class): The annotated bean or @Configuration class will only be loaded if a bean of type MyBean (or the specified name/qualifier) does not already exist in the Spring application context.

Use Case: This is crucial for Spring Boot's "opinionated defaults" while allowing user overrides. For example, Spring Boot might auto-configure a RestTemplate bean, but if the developer defines their own RestTemplate bean, Spring Boot's auto-configuration for RestTemplate will be skipped.

Crucial for Auto-configuration: These annotations enable Spring Boot to dynamically configure the application based on what dependencies are present and what the user has already provided. This allows for sensible defaults out-of-the-box (e.g., configure a web server if spring-boot-starter-web is present) while still providing flexibility for developers to override or customize any part of the auto-configuration.

What is @EnableAsync and @Async used for? How do you handle exceptions from @Async methods?

Answer:

@EnableAsync: This annotation, typically placed on a @Configuration class, enables Spring's asynchronous method execution capability. It activates Spring's proxying mechanism for methods annotated with @Async.

@Async: This annotation is placed on a method to indicate that it should be executed in a separate thread, asynchronously from the caller. The calling thread will not wait for the @Async method to complete.

Example:

Java

@Configuration
@EnableAsync
public class AsyncConfig {
    // Optionally define a custom TaskExecutor bean
}

@Service
public class EmailService {
    @Async
    public CompletableFuture<Void> sendEmail(String to, String subject, String body) {
        // Simulate long-running email sending
        System.out.println("Sending email in a separate thread...");
        return CompletableFuture.completedFuture(null);
    }
}
Exception Handling from @Async: Exceptions thrown by @Async methods are not propagated back to the calling thread. If a void method throws an exception, it will effectively be "swallowed" by the async executor.

Solution 1 (Returning Future/CompletableFuture): If the @Async method returns Future or CompletableFuture, exceptions thrown inside the async method will be wrapped in the Future and can be retrieved by the caller when get() is called on the Future.

Solution 2 (AsyncUncaughtExceptionHandler): For void @Async methods or to provide a global fallback, you can implement the AsyncUncaughtExceptionHandler interface and register it as a bean. Spring will then invoke this handler for unhandled exceptions from @Async methods.

Explain the purpose of @EnableCaching and caching-related annotations like @Cacheable, @CacheEvict, and @CachePut.

Answer: Spring Boot provides powerful caching abstraction to improve application performance by storing method results.

@EnableCaching: This annotation, placed on a @Configuration class, enables Spring's caching mechanism. It tells Spring to look for other caching annotations on beans.

@Cacheable:

Purpose: Marks a method whose results should be cached. If the cache already contains the result for the given arguments, the method is not executed, and the cached value is returned. Otherwise, the method is executed, its result is stored in the cache, and then returned.

Example: @Cacheable(value = "products", key = "#id") public Product getProductById(Long id)

@CacheEvict:

Purpose: Marks a method that should trigger a cache eviction. When this method is executed, one or more entries (or all entries) from the specified cache are removed.

Example: @CacheEvict(value = "products", key = "#id") public void deleteProduct(Long id) (removes a specific product)

@CacheEvict(value = "products", allEntries = true) (removes all products from the cache)

@CachePut:

Purpose: Marks a method whose results should be cached and always executed. The method's result is put into the cache, replacing any existing entry for the given key.

Example: @CachePut(value = "products", key = "#product.id") public Product updateProduct(Product product)

Use Case: Useful for update operations where you want the method to run (e.g., update the database) but also ensure the cache is updated with the latest value.

What is @Scheduled annotation used for? How would you configure its execution interval?

Answer: @Scheduled is a Spring annotation used to define methods that should be executed at fixed times or intervals. It's part of Spring's task scheduling capabilities.

Enabling: You need to add @EnableScheduling to one of your @Configuration classes to activate scheduled task execution.

Configuration:

fixedRate: Executes the task at a fixed rate, regardless of how long the previous execution took. The time is measured from the start of the previous execution.

Example: @Scheduled(fixedRate = 5000) (every 5 seconds)

fixedDelay: Executes the task at a fixed delay after the completion of the previous execution. The time is measured from the end of the previous execution.

Example: @Scheduled(fixedDelay = 5000) (5 seconds after the previous one finishes)

initialDelay: Specifies an initial delay before the first execution of a fixedRate or fixedDelay task.

Example: @Scheduled(initialDelay = 1000, fixedRate = 5000)

cron expression: Allows for more complex scheduling patterns using standard cron syntax.

Example: @Scheduled(cron = "0 0 9 * * MON-FRI") (every weekday at 9 AM)

Properties: Values can be externalized using SpEL and properties:

Example: @Scheduled(fixedRateString = "${my.scheduled.task.rate}")

When would you use @Validated in conjunction with other annotations like @RequestBody or @ConfigurationProperties?

Answer: @Validated is a Spring annotation (from org.springframework.validation.annotation.Validated) used to trigger JSR-303 (Bean Validation) on method parameters or on classes themselves.

Use Cases:

With @RequestBody (for REST API input validation): When a controller method receives a complex object in the request body, you can annotate that object parameter with @Valid (JSR-303 standard) or @Validated (Spring specific, allows grouping of validation constraints). Spring will then automatically validate the object against any JSR-303 annotations (@NotNull, @Size, @Email, @Min, @Max, etc.) present on its fields. If validation fails, MethodArgumentNotValidException is thrown.

Example: public ResponseEntity<Void> createUser(@Valid @RequestBody UserDTO userDto)

With @ConfigurationProperties (for configuration validation): You can annotate a @ConfigurationProperties class with @Validated. This ensures that when Spring binds external properties to the bean, it also performs validation checks on those properties. If the properties don't meet the specified constraints, the application will fail to start, preventing misconfigurations.

Security Annotations
What is the primary purpose of @EnableWebSecurity?
Answer:
@EnableWebSecurity is a Spring Security annotation that, when applied to a @Configuration class, enables Spring Security's web security support and provides the Spring Security configuration for web applications. It essentially imports the WebSecurityConfiguration and AuthenticationConfiguration classes, allowing you to customize security settings like authentication providers, authorization rules (e.g., URL access restrictions), form login, logout, and more, typically by extending WebSecurityConfigurerAdapter (though this is deprecated in newer Spring Security versions in favor of SecurityFilterChain beans).

Explain @PreAuthorize and @PostAuthorize. When would you use each?
Answer:
These are Spring Security method-level annotations used for fine-grained authorization checks. They leverage Spring Expression Language (SpEL) to evaluate security expressions.

@PreAuthorize: This annotation checks authorization before the method is executed. If the security expression evaluates to false, the method execution is prevented, and an AccessDeniedException is thrown.

Use Case: Most commonly used for checking permissions before an expensive operation or to ensure a user has the necessary roles/privileges to access a resource.

Example: @PreAuthorize("hasRole('ADMIN')") or @PreAuthorize("#userId == authentication.principal.id") (checks if the current user ID matches the requested user ID).

@PostAuthorize: This annotation checks authorization after the method has executed but before the result is returned to the caller. The security expression can access the method's return value (available as returnObject in SpEL).

Use Case: Less common than @PreAuthorize. Useful when the authorization decision depends on the properties of the object being returned by the method.

Example: @PostAuthorize("returnObject.owner == authentication.principal.username") (ensures the returned object's owner matches the current user).

What is @Secured and how does it compare to @PreAuthorize?
Answer:
@Secured is a simpler, role-based Spring Security annotation for authorization, primarily used for coarse-grained access control.

@Secured({"ROLE_ADMIN", "ROLE_USER"}): This annotation restricts method access based on whether the authenticated user has any of the specified roles. If the user doesn't have at least one of the roles, access is denied.

Comparison to @PreAuthorize:

Expressiveness: @PreAuthorize is much more powerful and flexible as it uses SpEL, allowing complex expressions involving method arguments, return values, and various security contexts. @Secured is limited to checking for specific roles.

Flexibility: @PreAuthorize can evaluate any boolean expression, while @Secured can only check for the presence of roles.

Usage: @PreAuthorize is generally preferred in modern Spring Security applications for its flexibility. @Secured is simpler for straightforward role checks but is often considered less flexible than SpEL-based annotations.

Dependency: For @Secured to work, you typically need to enable it via @EnableGlobalMethodSecurity(securedEnabled = true) (or similar configuration in newer versions). For @PreAuthorize, it's @EnableGlobalMethodSecurity(prePostEnabled = true).

Actuator and Monitoring Annotations (Less common for direct annotation Qs, but good to know)
How do you enable and configure Spring Boot Actuator endpoints?
Answer:

Enabling: To enable Spring Boot Actuator, you simply need to add the spring-boot-starter-actuator dependency to your project's pom.xml (for Maven) or build.gradle (for Gradle). Once added, Actuator endpoints are automatically discovered and available.

Configuration:

Exposure: By default, only the /health and /info endpoints are exposed over the web. To expose more (or all) endpoints, you configure the management.endpoints.web.exposure.include property in application.properties or application.yml.

management.endpoints.web.exposure.include=* (exposes all)

management.endpoints.web.exposure.include=health,info,env,beans (exposes specific ones)

Exclusion: You can also explicitly exclude endpoints using management.endpoints.web.exposure.exclude.

Base Path: The base path for Actuator endpoints can be customized using management.endpoints.web.base-path.

Security: Actuator endpoints are typically secured. You can integrate them with Spring Security to control access to specific endpoints based on roles or authentication.

Relevant Annotations (Indirectly related): While there aren't direct annotations on your code to enable Actuator, Spring Boot's internal auto-configuration uses @ConditionalOnClass to activate Actuator features when the starter is present.

Answer: @EnableRSocketClients enables scanning for RSocket client interfaces annotated with @RSocketClient. These annotations are used to declaratively define RSocket clients that can communicate with RSocket servers, abstracting away the RSocket communication details, similar to how @FeignClient works for REST.

Microservices & Cloud-Native Annotations (If Spring Cloud is present)

What is @EnableDiscoveryClient and why is it essential in a microservices architecture?

Answer: @EnableDiscoveryClient (or @EnableEurekaClient for Eureka specifically) enables a Spring Boot application to act as a discovery client. It allows the service instance to register itself with a Service Registry (e.g., Eureka, Consul, Zookeeper) and discover other registered services. It's essential for microservices because it enables dynamic service location and client-side load balancing without hardcoding service addresses.

What is @EnableFeignClients and how does @FeignClient simplify inter-service communication?

Answer:

@EnableFeignClients: Placed on a configuration class, it enables Feign client scanning for interfaces annotated with @FeignClient.

@FeignClient: Used on an interface, it declaratively defines a REST client for a specific microservice. Feign automatically generates the HTTP request boilerplate based on the interface methods and annotations (e.g., @GetMapping). This simplifies inter-service communication by making HTTP calls look like local method calls.

Explain @EnableCircuitBreaker (for Resilience4j/Hystrix) and @CircuitBreaker (from Resilience4j).

Answer:

@EnableCircuitBreaker (or @EnableResilience4j in older setups) enables circuit breaker functionality in your Spring Boot application (if using a library like Resilience4j).

@CircuitBreaker: Placed on a method, it wraps that method call in a circuit breaker. If the wrapped method repeatedly fails or times out, the circuit "opens," and subsequent calls are immediately short-circuited (fail fast) to prevent cascading failures. It typically requires a fallbackMethod to execute when the circuit is open.

What is @LoadBalanced used for in the context of WebClient.Builder or RestTemplate?

Answer: @LoadBalanced is applied to a RestTemplate or WebClient.Builder @Bean definition. It marks the instance as capable of client-side load balancing using a service discovery mechanism (e.g., Eureka/Consul with Spring Cloud LoadBalancer). When making a request to a service name (e.g., http://PRODUCT-SERVICE/products), the load balancer intercepts it and resolves the service name to an actual instance IP:Port from the service registry.

Other Useful Annotations

What is @TransactionalEventListener used for, and how does it relate to @EventListener?

Answer: @TransactionalEventListener is a specialization of @EventListener that ensures the event listener method is invoked only at specific phases of the transaction (e.g., after commit, after rollback, after completion). It's used to avoid issues where an event is processed prematurely before a transaction is committed, ensuring atomicity between a business operation and its resulting events.

Explain the purpose of @Retryable (from Spring Retry or Resilience4j).

Answer: @Retryable is placed on a method to indicate that if the method throws a specific exception, the call should be automatically retried a certain number of times before failing completely. It's used to make service calls more resilient to transient failures (e.g., temporary network glitches, brief service unavailability).
