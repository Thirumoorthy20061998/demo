class Vehicle {
    public void move() {
        System.out.println("Vehicle is moving.");
    }
}

// Subclass (Child)
class Bike extends Vehicle {
    // This method OVERRIDES the move() method in the Vehicle class
    @Override 
    public void move() {
        System.out.println("Bike is pedaled on the street.");
    }
}

public class OverridingExample {
    public static void main(String[] args) {
        Vehicle myVehicle = new Vehicle();
        Bike myBike = new Bike();

        System.out.println("--- Method Overriding Example ---");
        
        // Calls the Vehicle's move()
        myVehicle.move(); // Output: Vehicle is moving.
        
        // Calls the Bike's move() (the overridden version)
        myBike.move();    // Output: Bike is pedaled on the street.
    }
}


class Printer {
    
    // 1. Overloaded method: prints an integer
    public void print(int data) {
        System.out.println("Printing an integer: " + data);
    }

    // 2. Overloaded method: prints a string (Different TYPE of parameter)
    public void print(String data) {
        System.out.println("Printing a string: " + data);
    }

    // 3. Overloaded method: prints an integer and a string (Different NUMBER and TYPES)
    public void print(int number, String label) {
        System.out.println("Printing labeled data: " + label + " - " + number);
    }
}

public class OverloadingExample {
    public static void main(String[] args) {
        Printer p = new Printer();

        System.out.println("\n--- Method Overloading Example ---");
        
        // Calls print(int)
        p.print(101);             // Output: Printing an integer: 101
        
        // Calls print(String)
        p.print("Hello Java!");  // Output: Printing a string: Hello Java!
        
        // Calls print(int, String)
        p.print(25, "Count");     // Output: Printing labeled data: Count - 25
    }
}






Encapsulation in Inheritance (Java)
1. The Superclass (DataHolder) - Enforcing Encapsulation
The parent class hides its data (secretNumber) using private. The only way to read the number is through the public method, getSecretNumber().

Java

class DataHolder {
    // Encapsulated data: 'private' means no direct access, even by child classes
    private int secretNumber = 42; 

    // Public method to allow controlled access (Getter)
    public int getSecretNumber() {
        return secretNumber;
    }
}
2. The Subclass (Accessor) - Respecting Encapsulation
The child class inherits from DataHolder but cannot directly access secretNumber. It must use the inherited public method (getSecretNumber()).

Java

class Accessor extends DataHolder {
    
    public void displaySecret() {
        // ERROR: If you uncomment the line below, the code will fail to compile.
        // System.out.println(secretNumber); // Compiler Error: 'secretNumber' has private access

        // SUCCESS: Must use the public getter method from the parent.
        int value = getSecretNumber();
        System.out.println("Child class accessed the secret number: " + value);
    }
}

public class SimpleEncapsulationInheritanceDemo {
    public static void main(String[] args) {
        Accessor a = new Accessor();
        
        System.out.println("--- Encapsulation Maintained ---");
        a.displaySecret(); 
    }
}

3. Provide a simple Java code example that demonstrates a Deadlock.

Answer: A classic example involves two threads trying to acquire two locks in opposite orders.

Java

public class DeadlockExample {
    private static Object lock1 = new Object();
    private static Object lock2 = new Object();

    public static void main(String[] args) {
        // Thread 1
        Thread thread1 = new Thread(() -> {
            synchronized (lock1) {
                System.out.println("Thread 1: Acquired lock1");
                try {
                    Thread.sleep(100); // Simulate some work
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                System.out.println("Thread 1: Waiting for lock2...");
                synchronized (lock2) {
                    System.out.println("Thread 1: Acquired lock2");
                }
            }
        });

        // Thread 2
        Thread thread2 = new Thread(() -> {
            synchronized (lock2) {
                System.out.println("Thread 2: Acquired lock2");
                try {
                    Thread.sleep(100); // Simulate some work
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
                System.out.println("Thread 2: Waiting for lock1...");
                synchronized (lock1) {
                    System.out.println("Thread 2: Acquired lock1");
                }
            }
        });

        thread1.start();
        thread2.start();
    }
}
Explanation:

Thread 1 acquires lock1, then tries to acquire lock2.

Thread 2 acquires lock2, then tries to acquire lock1.

If Thread 1 acquires lock1 and then the scheduler switches to Thread 2, which acquires lock2, both threads will then be stuck waiting for the lock held by the other, resulting in a deadlock.

// import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;

@Service
public class ProductServiceClient {

    private final RestTemplate restTemplate; // or WebClient

    public ProductServiceClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    // Define a name for your circuit breaker instance
    private static final String PRODUCT_SERVICE_CB = "productService";

    @CircuitBreaker(name = PRODUCT_SERVICE_CB, fallbackMethod = "getProductFallback")
    public Product getProductDetails(Long productId) {
        System.out.println("Attempting to get product " + productId);
        // Simulate a call that might fail
        if (productId % 2 == 0) { // For demonstration, fail on even IDs
            throw new RuntimeException("Simulated product service error for ID: " + productId);
        }
        // Actual RestTemplate call
        // return restTemplate.getForObject("http://product-service/products/" + productId, Product.class);
        return new Product(productId, "Real Product " + productId, 100.0);

    // Fallback method must have the same signature as the original (or compatible)
    public Product getProductFallback(Long productId, Throwable t) {

        return new Product(productId, "Fallback Product", 0.0);
    }

"Explain the Producer-Consumer problem and how this code solves it. Specifically, how do the wait() and notify() methods prevent race conditions and ensure proper synchronization between the producer and consumer threads?"





import java.util.LinkedList;
import java.util.List;

class ProducerConsumerExample {
    public static void main(String[] args) {
        SharedBuffer buffer = new SharedBuffer(5); // Buffer size of 5
        Thread producerThread = new Thread(new Producer(buffer));
        Thread consumerThread = new Thread(new Consumer(buffer));
        
        producerThread.start();
        consumerThread.start();
    }
}

class SharedBuffer {
    private List<Integer> buffer = new LinkedList<>();
    private final int capacity;

    public SharedBuffer(int capacity) {
        this.capacity = capacity;
    }

    public void produce(int item) throws InterruptedException {
        synchronized (this) {
            while (buffer.size() == capacity) {
                System.out.println("Buffer is full. Producer waiting...");
                wait(); // Producer waits if buffer is full
            }
            buffer.add(item);
            System.out.println("Produced: " + item);
            notify(); // Notifies the waiting consumer
        }
    }

    public int consume() throws InterruptedException {
        synchronized (this) {
            while (buffer.isEmpty()) {
                System.out.println("Buffer is empty. Consumer waiting...");
                wait(); // Consumer waits if buffer is empty
            }
            int item = buffer.remove(0);
            System.out.println("Consumed: " + item);
            notify(); // Notifies the waiting producer
            return item;
        }
    }
}

class Producer implements Runnable {
    private final SharedBuffer buffer;

    public Producer(SharedBuffer buffer) {
        this.buffer = buffer;
    }

    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            try {
                buffer.produce(i);
                Thread.sleep(100); // Simulating work
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}

class Consumer implements Runnable {
    private final SharedBuffer buffer;

    public Consumer(SharedBuffer buffer) {
        this.buffer = buffer;
    }

    @Override
    public void run() {
        for (int i = 0; i < 10; i++) {
            try {
                buffer.consume();
                Thread.sleep(200); // Simulating work

Explain the concept of a thread pool and the benefits of using ExecutorService. How would you create a fixed-size thread pool and submit a task that returns a value?"

The Answer:
Thread Pool: A thread pool is a managed collection of pre-initialized threads that can be reused to execute tasks. Instead of creating a new thread for every task, a thread pool reuses existing threads, which reduces the overhead of thread creation and destruction.

Benefits of ExecutorService: It's a high-level API for managing thread pools and executing tasks. Its benefits include:

Reduced Overhead: By reusing threads, it minimizes the performance cost of creating and destroying threads.

Resource Management: It provides control over the number of threads, preventing the application from being overwhelmed by too many threads.

Task Management: It decouples the task from its execution, simplifying code.

Future and Callable: It allows you to submit tasks that can return a value or throw a checked exception, which is not possible with the Runnable interface.
